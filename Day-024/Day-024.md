## 0. 들어가며 (Motivation)

[[Day-007]] 에서 우리는 문서를 **단어 개수 벡터(BoW)** 로, 그리고 **TF-IDF** 로 "흔한 단어는 깎고 변별력 있는 단어는 키운" 벡터로 바꾸는 법을 배웠습니다. [[Day-004]] 에서는 두 벡터가 얼마나 "같은 방향"인지를 재는 **코사인 유사도** 를 익혔죠. 오늘은 이 둘을 **하나의 검색 모델로 조립** 합니다.

아이디어는 놀랄 만큼 단순하고 강력합니다:

> **벡터 공간 모델(VSM)**: 모든 문서와 질의를 **같은 어휘 축(axis)을 가진 벡터**로 놓는다. 각 축은 하나의 단어, 그 값은 그 단어의 TF-IDF 가중치. 그러면 "질의와 가장 비슷한 문서" = "질의 벡터와 **각도가 가장 작은**(코사인이 가장 큰) 문서" 다.

[[Day-017]] 의 BM25 가 *확률·적합성* 이라는 렌즈로 랭킹을 봤다면, VSM 은 *기하학(geometry)* 이라는 렌즈로 봅니다. 둘은 출발점이 완전히 다른데도 실전에서 비슷한 순위를 냅니다 — 오늘 (5)에서 직접 확인합니다. 그리고 VSM 의 "문서를 한 공간 속 점(벡터)으로 본다"는 발상은, 단어 개수 대신 **학습된 의미 벡터(임베딩)** 를 쓰는 순간 그대로 Phase 5 의 **밀집 검색(dense retrieval)** 이 됩니다. 오늘은 그 문을 여는 날입니다.

> IR has two classic ranking pillars: BM25 (from probabilistic relevance) and the **Vector Space Model** — represent documents and the query as TF-IDF vectors and rank by **cosine similarity** (smallest angle). VSM's "document as a point in space" view becomes Phase 5's dense retrieval once counts are swapped for learned embeddings.

## 1. 핵심 개념 (Core Concepts)

### 1.1 문서를 벡터로 — 공통 어휘 공간 (term-document space)

어휘(vocabulary)가 $V$개 단어라면, 모든 문서·질의는 **$V$차원 공간의 한 벡터**입니다. 축 $t$의 좌표는 그 문서에서 단어 $t$의 **가중치**(대개 TF-IDF). 대부분의 단어는 한 문서에 안 나오므로 이 벡터는 **희소(sparse)** 합니다.

- 문서 $d \to \vec{d} = (w_{1,d}, w_{2,d}, \dots, w_{V,d})$
- 질의 $q \to \vec{q} = (w_{1,q}, \dots, w_{V,q})$

이제 "관련성"은 **두 벡터의 기하학적 가까움** 문제로 바뀝니다.

### 1.2 왜 유클리드 거리가 아니라 코사인인가 ([[Day-004]])

두 벡터의 가까움을 잴 때 **유클리드 거리**를 쓰면 안 될까요? 문제는 **문서 길이**입니다. 같은 주제라도 긴 문서는 단어가 많아 벡터의 **크기(norm)** 가 큽니다. 유클리드 거리는 이 크기 차이에 휘둘려, *내용은 같은데 길이만 다른* 문서를 "멀다"고 판단합니다.

**코사인 유사도**는 크기를 버리고 **방향(각도)만** 봅니다:

$$
\cos(\vec{q}, \vec{d}) = \frac{\vec{q}\cdot\vec{d}}{\lVert\vec{q}\rVert\,\lVert\vec{d}\rVert}
= \frac{\sum_t w_{t,q}\,w_{t,d}}{\sqrt{\sum_t w_{t,q}^2}\,\sqrt{\sum_t w_{t,d}^2}}
$$

분자는 **내적(dot product)** — 두 벡터가 공유하는 단어들의 가중치 곱의 합. 분모의 $\lVert\cdot\rVert$가 바로 **길이 정규화**: 벡터를 단위 길이로 만들어 "길다고 유리"한 편향을 지웁니다. 값은 $[0,1]$(가중치가 음수가 아니므로) — 1이면 완전히 같은 방향, 0이면 공유 단어 없음.

### 1.3 TF-IDF 가중 — 벡터의 좌표값 정하기 ([[Day-007]])

축의 값을 무엇으로 채울까? 단순 개수(raw tf)는 문제가 많습니다(흔한 단어가 지배). 오늘 코드가 쓰는 표준 조합은 **로그 tf × idf** 입니다:

$$
w_{t,d} = \underbrace{(1 + \log_{10}\text{tf}_{t,d})}_{\text{로그 tf: 반복은 체감}} \times \underbrace{\log_{10}\frac{N}{\text{df}_t}}_{\text{idf: 흔한 단어 깎기}}
$$

- **로그 tf**: 단어가 10번 나온다고 1번의 10배 중요하진 않다 → 로그로 눌러 **포화(saturation)**. (BM25 의 tf 포화와 같은 동기)
- **idf**: 모든 문서에 나오는 단어(df=N)는 $\log(N/N){=}0$ → **변별력 없는 단어 제거**.

> [!note] SMART 표기 — `ltc.ltc`, `lnc.ltc`?
> IR 교과서는 (문서가중.질의가중) 을 세 글자 코드로 씁니다: **l**(log tf)·**t**(idf)·**c**(cosine 정규화). 오늘 코드는 문서·질의 **양쪽 모두 `ltc`**(log-tf·idf·cosine)를 씁니다. 실무에선 문서는 idf 생략(`lnc`), 질의만 idf 적용(`ltc`)하는 **`lnc.ltc`** 가 흔합니다 — idf 를 한 번만 곱해 이중 반영을 피하려는 것.

### 1.4 랭킹 절차 (한눈에)

1. 코퍼스로 **df·idf** 계산(색인 시).
2. 각 문서를 **TF-IDF 벡터**로 만들고 (원한다면 미리 정규화해 저장).
3. 질의가 오면 질의도 **TF-IDF 벡터**로.
4. 모든 문서에 대해 **코사인** 계산 → 내림차순 정렬 → top-k.

BM25 와 달리 VSM 은 질의어가 없어도 문서 벡터는 그대로라, **어떤 유사도 기준이든**(코사인·내적) 갈아 끼울 수 있는 게 장점입니다.

### 1.5 내적 vs 코사인 — 정규화가 만드는 차이

분자(내적)만 쓰면 어떻게 될까요? 내적은 $\lVert\vec{d}\rVert$에 비례하므로 **긴 문서·단어를 많이 반복한 문서가 유리** 해집니다. 오늘 실습의 `d13`("vector" 를 5번 반복한 긴 잡문)이 그 함정 문서 — **내적 랭킹에선 3위**로 치고 올라오지만, **코사인에선 4위**로 내려갑니다. 딱 한 줄 차이(분모의 $\lVert\vec{d}\rVert$)가 스팸성 긴 문서를 눌러 준 것입니다.

## 2. 직관 / 비유 (Intuition)

> [!tip] 비유 — 나침반의 방향, 지도상의 거리
> 두 등산객이 같은 봉우리를 가리키는지 알고 싶습니다. 한 명은 산 코앞에, 한 명은 멀리 있어도, **손가락이 같은 방향** 을 가리키면 "같은 목표"죠. 코사인은 이 **방향(각도)** 만 봅니다 — 거리(문서 길이)는 무시.
> 반대로 "누가 더 많이 외쳤나"(내적·유클리드)로 판단하면, 목이 터져라 같은 단어를 반복하는 **수다쟁이 긴 문서** 가 이깁니다. 코사인은 "얼마나 크게 말했나"가 아니라 "어느 쪽을 가리키나"를 재기에 그 속임수에 안 넘어갑니다.
>
> Cosine cares about the *direction* two vectors point (same topic?), not their *magnitude* (document length / repetition) — so a long, keyword-stuffed document can't win just by shouting.

```text
                               ↑ 내적(길이)은 크지만 방향이 어긋나 cos 낮음(0.13)

   BM25 는 같은 목표(길이 보정·tf 포화)를 '확률' 언어로, VSM 은 '기하' 언어로 한다.
```

> Documents and the query are points in a term-weighted space; rank by the angle to the query. The `||d||` denominator cancels a long document's size advantage — the same goal BM25 reaches through probabilistic length normalization.

### 4.2 VSM 핵심 — `vsm_cosine.py` (발췌)

```python
# (발췌) vsm_cosine.py — Day-024
# ② TF-IDF 벡터: w = (1+log10 tf) * idf   (log-frequency weighting)
def tfidf_vector(ix, toks):
    tf = Counter(toks)
    vec = {}
    for t, f in tf.items():
        idf = ix.idf(t)              # log10(N/df); df=N 이면 0 → 자동 제거
        if idf == 0.0:
            continue
        vec[t] = (1.0 + math.log10(f)) * idf
    return vec

def dot(a, b):                        # 희소 벡터 내적 (공유 단어만)
    small, big = (a, b) if len(a) <= len(b) else (b, a)
    return sum(w * big.get(t, 0.0) for t, w in small.items())

def norm(v):
    return math.sqrt(sum(w * w for w in v.values()))

# ③ 코사인 = 내적 / (||a||·||b||)  ← 분모가 '길이 정규화'
def cosine(a, b):
    na, nb = norm(a), norm(b)
    if na == 0 or nb == 0:
        return 0.0
    return dot(a, b) / (na * nb)
```

### 4.3 결과를 정직하게 읽기 (관찰 포인트)

1. **(1) 질의 벡터의 좌표가 곧 '중요도'다.** `model`(df=1, idf=1.11)이 가장 무겁고, 흔할수록(`vector` df=3) 가벼워집니다 — [[Day-007]] TF-IDF 가 벡터 좌표로 그대로 들어간 모습입니다. 이 벡터의 길이 $\lVert q \rVert=1.7228$ 이 코사인 분모의 한 축입니다.
2. **(2) 코사인이 '가장 같은 방향'을 고른다.** 세 단어(vector·space·model)를 모두 담은 `d11` 이 0.5872 로 압도적 1위, "cosine·query·vector" 를 가진 `d12` 가 2위. 주제어를 짧은 문서에 밀도 높게 담을수록 코사인이 큽니다.
3. **(3)↔(2) 내적은 순위를 바꾼다.** 정규화를 빼면 `d13`(긴 잡문)이 **3위로 상승** — `vector` 를 5번 반복해 내적(0.6890)을 부풀렸기 때문입니다. 코사인에선 이 `d13` 이 **4위**(0.1332)로 밀려납니다. **딱 그 한 계단이 길이 정규화의 값어치** 입니다.
4. **(4) 숫자로 못 박기.** `d13` 은 길이 12(최장)이고 `dot=0.6890` 으로 `d10`(0.6608)보다 큰데, 코사인은 `d13`(0.1332) < `d10`(0.1391) 로 뒤집힙니다. 분모 $\lVert d \rVert$ 가 큰 `d13` 을 더 세게 깎은 것 — "많이 말한 것"이 아니라 "무엇을 가리키나"를 본 결과입니다.
5. **(5) BM25 는 다른 길로 비슷한 곳에.** 확률 모델 BM25 도 top-2 는 `d11·d12` 로 VSM 과 같습니다. 단 BM25 는 `d13` 을 여전히 3위에 둡니다 — tf 포화·길이 정규화가 `d13` 의 반복을 눌러 주긴 하나, 코사인만큼 강하게 강등하진 않습니다. **두 모델은 목표(길이·빈도 보정)가 같아도 강도와 방식이 다르다** 는 걸 보여 줍니다.

### 4.4 (실험) 직접 흔들어 보기

- **질의 바꾸기**: `query = "ranking metric"` 이나 `"index query"` 로 바꿔 코사인·BM25 순위가 어떻게 갈리는지 보세요.
- **문서 정규화 방식(`lnc.ltc`)**: 문서 벡터에서 idf 를 빼고(=raw log-tf) 질의에만 idf 를 두면 순위가 바뀌는지 확인해 보세요(실무 기본값).
- **`d13` 더 부풀리기**: `d13` 의 "vector" 반복을 10개로 늘려 보세요. 내적 순위는 더 오르지만 코사인은 거의 그대로임을 관찰하세요.
- **유클리드 거리와 비교**: `cosine` 대신 두 벡터의 유클리드 거리로 랭킹해, 왜 IR 이 코사인을 택했는지 체감해 보세요.

## 6. 생각거리 (Reflection)

> [!question] 스스로 답해 보기
> 1. **왜 코사인인가?** 유클리드 거리로 랭킹하면 같은 주제의 긴 문서/짧은 문서가 어떻게 취급될까요? (3)·(4)의 `d13` 수치로, 코사인이 그 문제를 어떻게 푸는지 설명해 보세요.
> 2. **BM25 vs VSM.** (5)에서 두 모델의 top-2 는 같은데 `d13` 처리에서 갈렸습니다. BM25 의 tf 포화(k1)·길이 정규화(b)와 코사인의 분모 $\lVert d\rVert$ 는 각각 "무엇을" 보정하나요? 둘 다 결국 무엇을 막으려는 걸까요?
> 3. **VSM 의 근본 한계.** VSM 도 결국 **글자가 겹치는 단어**들의 가중치 내적입니다. 그렇다면 [[Day-023]] 의 어휘 불일치(car↔automobile)를 VSM 은 풀 수 있을까요? 이 한계가 왜 Phase 5 의 **밀집 임베딩**([[Day-008]])으로 이어지는지 연결해 보세요.

## 7. 오늘의 한 줄 요약 (Takeaway)

> [!summary]
> **벡터 공간 모델(VSM)** 은 IR 의 또 다른 고전 기둥으로, 문서와 질의를 **TF-IDF 가중 벡터**([[Day-007]])로 놓고 그 사이의 **코사인 유사도**([[Day-004]])로 순위를 매긴다. 코사인 = 내적 ÷ (‖q‖·‖d‖) 이며, **분모의 길이 정규화가 핵심** — "많이·길게 반복한 문서"의 부당한 이득을 지워 *방향(주제 일치)* 만 본다. 오늘 표준 라이브러리만으로 VSM 을 직접 만들어, 질의 `"vector space model cosine"` 에 대해 코사인이 `d11`(0.5872)·`d12`(0.3032)를 상위로 올리는 걸 확인했고, **정규화를 뺀 내적**에선 긴 잡문 `d13` 이 3위로 부풀지만 **코사인에선 4위로 강등**됨(dot 0.6890 vs cos 0.1332)을 수치로 못 박았다. 같은 질의를 **BM25 로도 돌려**, 두 모델이 다른 이론(기하 vs 확률)에서 출발해도 top-2 가 일치하되 `d13` 처리 강도가 다름을 봤다. VSM 의 "문서 = 공간 속 한 점" 이라는 관점은, 단어 개수를 **학습된 의미 벡터**로 바꾸는 순간 그대로 **Phase 5 의 밀집 검색**([[Day-008]])이 된다 — 오늘은 그 문을 연 셈이다.
