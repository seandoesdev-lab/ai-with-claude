## 0. 들어가며 (Motivation)

[[Day-016]] 으로 **좁히고**, [[Day-017]] 의 BM25 로 **줄 세웠습니다**. 두 질의 함수를 바꿔 가며 "이게 더 좋아 보인다" 고 말할 수는 있지만 — *눈대중* 입니다. `k1=1.5` 가 나은가 `2.0` 이 나은가? BM25 가 [[Day-015]] 의 TF-IDF 코사인보다 정말 나은가? 이런 질문에 **재현 가능한 숫자** 로 답하지 못하면 검색 시스템을 *개선* 할 수 없습니다. 측정할 수 없으면 개선할 수 없으니까요.

[[Day-005]] 에서 우리는 분류기를 **정밀도(precision)·재현율(recall)·F1** 로 평가했습니다. 하지만 그 지표들은 "맞다/틀리다" 의 **집합** 만 봅니다 — 순서가 없죠. 검색은 다릅니다. 검색의 출력은 **순위가 매겨진 목록(ranked list)** 이고, *같은 문서들을 찾았어도 순서가 다르면 사용자 경험은 천지차이* 입니다. 정답을 1등에 둔 검색과 10등에 둔 검색은 똑같이 "찾긴 했다" 지만 결코 같은 품질이 아닙니다.

그래서 IR 평가는 [[Day-005]] 의 정밀도·재현율을 **순위를 의식하도록** 확장합니다. 오늘 배울 다섯 지표는 정확히 이 "순서까지 보는 평가" 의 표준 도구입니다.

> Day-016 narrowed, Day-017 ranked — but *how good* is the ranking? We extend Day-005's precision/recall to **ranked** retrieval. The output is an ordered list, so order matters: the same hits in a different order is not the same quality. Today's five metrics are the standard tools for *order-aware* evaluation.

## 1. 핵심 개념 (Core Concepts)

### 1.1 관련성 판단(qrels) — 평가의 정답지

어떤 지표든 **정답** 이 있어야 채점합니다. IR 에서 이 정답을 **관련성 판단(relevance judgments, 흔히 `qrels`)** 이라 부릅니다: *질의마다 어떤 문서가 관련 있는지* 를 사람이(또는 클릭 로그로) 표시한 것입니다.

- **이진(binary)**: 관련 / 비관련 (1 / 0).
- **등급(graded)**: 0~3 처럼 *얼마나* 관련 있는지 단계로 (3=완벽, 1=약간, 0=무관). nDCG 가 이걸 씁니다.

> 오늘은 qrels 가 *주어졌다고* 보고 지표 계산에 집중합니다. 이 정답지를 *현실에서 어떻게 모으는가*(테스트 컬렉션·pooling)는 다음 편([[Day-019]])의 주제입니다.

### 1.2 Precision@k 와 Recall@k — [[Day-005]] 를 컷오프로

사용자는 보통 **상위 몇 개** 만 봅니다(구글 1페이지). 그래서 IR 정밀도·재현율은 **상위 k개로 잘라서(@k)** 잽니다.

$$ \text{Precision@}k = \frac{\text{상위 }k\text{개 중 관련 문서 수}}{k}, \qquad \text{Recall@}k = \frac{\text{상위 }k\text{개 중 관련 문서 수}}{\text{전체 관련 문서 수 }R} $$

- **Precision@k**: "위에 보여준 것 중 얼마나 알짜인가" (사용자 체감 직결).
- **Recall@k**: "있는 정답 중 얼마나 위로 건졌나" (놓친 게 얼마나 적은가).
- 둘은 [[Day-005]] 의 정밀도·재현율과 *똑같은 정신* 이되, **순위 컷오프 k** 가 더해졌을 뿐입니다.

> [!warning] @k 의 한계
> Precision@k 는 *상위 k개 안의 순서* 는 보지 않습니다. 관련 문서가 1등이든 k등이든 P@k 는 같죠. "순서까지" 보려면 아래 MAP·MRR·nDCG 가 필요합니다.

### 1.3 Average Precision(AP) 과 MAP — 관련 문서를 만날 때마다 채점

**평균 정밀도(Average Precision, AP)** 는 순위를 위에서부터 훑다가 **관련 문서를 만날 때마다 그 위치의 Precision@i 를 기록** 하고, 그것들을 평균한 값입니다:

$$ \text{AP} = \frac{1}{R}\sum_{i:\,\text{관련}} \text{Precision@}i $$

($R$ = 전체 관련 문서 수.) 관련 문서가 **위에 몰려 있을수록** 각 시점의 Precision@i 가 높아 AP 가 커집니다. AP 는 *한 질의* 의 점수이고, 여러 질의의 AP 를 평균하면 **MAP(Mean Average Precision)** — IR 에서 가장 널리 쓰이는 단일 요약 지표입니다.

> [!tip] AP 의 직관
> "관련 문서 하나하나를, *그 지점까지의 정밀도* 로 평가해 평균" → 관련 문서를 **얼마나 일찍·빽빽이** 끌어올렸는지를 한 숫자로 압축한다.

### 1.4 MRR — 첫 정답이 얼마나 위에 있나

질의응답·"느낌 오는 첫 결과" 가 중요한 검색에서는 **첫 번째 관련 문서의 순위** 만 봐도 충분할 때가 많습니다. **역순위(Reciprocal Rank, RR)** 는 그 순위의 역수입니다:

$$ \text{RR} = \frac{1}{\text{첫 관련 문서의 순위}}, \qquad \text{MRR} = \frac{1}{|Q|}\sum_{q} \text{RR}_q $$

1등이면 1.0, 2등이면 0.5, 5등이면 0.2. 여러 질의에 대해 평균하면 **MRR(Mean Reciprocal Rank)** 입니다. *첫 정답까지 몇 칸 내려가야 하나* 를 재는, 단순하지만 강력한 지표입니다.

### 1.5 nDCG — 관련도의 *등급* 과 *위치* 를 함께

앞 지표들은 관련을 0/1 로만 봤습니다. 하지만 "완벽한 답(3)" 과 "약간 관련(1)" 은 다르게 대접받아야 합니다. **nDCG(normalized Discounted Cumulative Gain)** 가 이걸 다룹니다 — *등급(graded relevance)* 과 *위치 할인(position discount)* 을 동시에.

- **DCG@k**: 각 문서의 관련 등급을 *순위로 할인* 해 더합니다. 아래로 갈수록 $\log_2$ 로 가치를 깎습니다:

$$ \text{DCG@}k = \sum_{i=1}^{k} \frac{rel_i}{\log_2(i+1)} $$

- **IDCG@k**: *이상적 순위* (등급 높은 순으로 완벽히 정렬)의 DCG — 달성 가능한 최댓값.
- **nDCG@k** $= \text{DCG@}k / \text{IDCG@}k$ → 0~1 로 정규화. 1.0 이면 "이상적 순위와 동일".

$$ \text{nDCG@}k = \frac{\text{DCG@}k}{\text{IDCG@}k} $$

정규화 덕분에 *관련 문서 수가 다른 질의끼리도* 공정하게 평균할 수 있습니다. 그래서 nDCG 는 **웹 검색·추천의 사실상 표준** 입니다.

> [!note] 이득(gain) 두 가지 변형
> 위 식은 *선형 이득(linear gain, $rel_i$)* 입니다. 등급 차이를 더 크게 벌리려고 *지수 이득* $\;(2^{rel_i}-1)$ 을 쓰는 변형(Burges 2005)도 흔합니다. 오늘 코드는 직관이 잘 보이는 선형 이득을 씁니다.

## 2. 직관 / 비유 (Intuition)

> [!tip] 비유 — 시험 채점 vs 등수 매기기
> [[Day-005]] 의 정밀도·재현율은 **OX 채점** 같습니다 — 맞은 개수만 셉니다. 그런데 검색은 **달리기 등수** 에 가깝습니다: 누가 결승선을 통과했나(재현율)뿐 아니라 *몇 등으로* 통과했나가 중요하죠.
>
> - **Precision@k** = "상위 k명 중 진짜 실력자 비율".
> - **MRR** = "1등(첫 정답)이 출발선에서 몇 번째에 있었나" — 빨리 나올수록 점수↑.
> - **MAP** = "실력자들이 *얼마나 앞쪽에 빽빽이* 들어왔나"를 한 숫자로.
> - **nDCG** = 등수에 더해 *선수의 등급(금·은·동)* 까지 반영 — 금메달리스트를 1등에 둬야 만점, 동메달리스트를 1등에 두면 감점.
>
> 핵심: **같은 선수들이 들어와도(=같은 recall) 등수 배치가 다르면 점수가 다르다.** 이것이 집합 지표(precision/recall)와 순위 지표(MAP/nDCG)의 갈림길입니다.
>
> Precision/recall = OX grading (count hits). Ranked metrics = race placement: not just *who* finished, but *in what order* and *what medal*. Same finishers in a different order ⇒ different MAP/nDCG, identical recall.

## 3. 한 장의 그림 (One Picture)

```text
  순위(상위→하위)   관련?  등급      각 지표가 보는 것
  ┌──────────────┬──────┬──────┐
  1. d2           ✔     2     P@1=1/1   ← Precision@k: 위 k칸 중 ✔ 비율
  2. d1           ✔     3     P@2=2/2      (1·2 줄만, 3줄부터는 무시)
  3. d3           ✗     0
  4. d4           ✔     1     P@4=3/4   ← AP: ✔ 만난 칸의 P@i 평균
  5. d6           ✗     0
  └──────────────┴──────┴──────┘
       │              │
       │              └─ MRR: 첫 ✔(d2)이 1등 → RR=1/1=1.0
       │
       └─ nDCG: Σ rel_i / log2(i+1)  를  이상적 순위(3,2,1,…)의 값으로 나눔
              위로 갈수록 분모 log2 ↑ → '아래에 둔 관련 문서'는 값이 깎인다

   집합 지표(P@k·R@k): "찾았나"          ← 순서 거의 안 봄
   순위 지표(MAP·MRR·nDCG): "어디에 뒀나"  ← 순서가 점수를 바꾼다
```

> Set metrics ask *whether* relevant docs were found; ranked metrics ask *where* they were placed. Putting a relevant doc lower costs DCG via the `log2` discount.

### 4.2 다섯 지표를 직접 만들기 — `ir_metrics.py`

아래를 **`ir_metrics.py`** 로 저장합니다(이 노트 옆에 함께 제공). 1절의 수식이 그대로 함수가 됩니다. 데이터는 [[Day-017]] BM25 가 반환했을 법한 두 질의의 순위와, 각 질의의 **등급 qrels(0~3)** 입니다.

```python
# ir_metrics.py — Day-018: IR 평가지표 from scratch (표준 라이브러리만)
# Precision@k · Recall@k · Average Precision(MAP) · MRR · nDCG
# 실행: uv run python ir_metrics.py
import sys, math
sys.stdout.reconfigure(encoding="utf-8")  # Windows 콘솔 한글 깨짐 방지

# 0) 관련성 판단(qrels) — 등급(graded) 0~3. 0 은 비관련(목록에 없으면 0).
qrels = {
    "Q1": {"d1": 3, "d2": 2, "d4": 1},   # d1 가장 관련, d4 약하게 관련
    "Q2": {"d3": 3, "d5": 1},
}
# 시스템(예: Day-017 BM25)이 반환한 순위 — 상위→하위
rankings = {
    "Q1": ["d2", "d1", "d3", "d4", "d6"],
    "Q2": ["d6", "d3", "d1", "d5", "d2"],
}

def rel(q, doc):       return qrels[q].get(doc, 0)        # 관련 등급(없으면 0)
def is_rel(q, doc):    return rel(q, doc) > 0             # 이진 관련성
def n_relevant(q):     return sum(1 for g in qrels[q].values() if g > 0)  # R

def precision_at_k(q, ranking, k):
    return sum(is_rel(q, d) for d in ranking[:k]) / k
def recall_at_k(q, ranking, k):
    R = n_relevant(q)
    return (sum(is_rel(q, d) for d in ranking[:k]) / R) if R else 0.0

def average_precision(q, ranking):           # AP (MAP 은 AP 의 질의 평균)
    R = n_relevant(q)
    if R == 0: return 0.0
    hits, score = 0, 0.0
    for i, d in enumerate(ranking, 1):       # 관련 문서 만날 때마다 P@i 누적
        if is_rel(q, d):
            hits += 1
            score += hits / i
    return score / R
def reciprocal_rank(q, ranking):             # RR (MRR 은 RR 의 질의 평균)
    for i, d in enumerate(ranking, 1):
        if is_rel(q, d): return 1.0 / i
    return 0.0

def dcg_at_k(q, ranking, k):                 # 선형 이득: grade / log2(rank+1)
    return sum(rel(q, d) / math.log2(i + 1) for i, d in enumerate(ranking[:k], 1))
def ndcg_at_k(q, ranking, k):
    dcg = dcg_at_k(q, ranking, k)
    ideal = sorted(qrels[q].values(), reverse=True)[:k]      # 이상적 순위
    idcg = sum(g / math.log2(i + 1) for i, g in enumerate(ideal, 1))
    return (dcg / idcg) if idcg else 0.0

print("=== 질의별 지표 ===")
aps, rrs, ndcgs = [], [], []
for q in ["Q1", "Q2"]:
    r = rankings[q]
    grades = ", ".join(f"{d}={g}" for d, g in qrels[q].items())
    print(f"\n[{q}] ranking={r}  (관련 grade: {grades})")
    print(f"  P@1={precision_at_k(q,r,1):.3f}  P@3={precision_at_k(q,r,3):.3f}  P@5={precision_at_k(q,r,5):.3f}")
    print(f"  R@3={recall_at_k(q,r,3):.3f}  R@5={recall_at_k(q,r,5):.3f}")
    ap, rr, nd = average_precision(q,r), reciprocal_rank(q,r), ndcg_at_k(q,r,5)
    print(f"  AP={ap:.4f}   RR={rr:.4f}   nDCG@5={nd:.4f}")
    aps.append(ap); rrs.append(rr); ndcgs.append(nd)

print("\n=== 컬렉션 전체 (질의 평균) ===")
print(f"  MAP    = {sum(aps)/len(aps):.4f}")
print(f"  MRR    = {sum(rrs)/len(rrs):.4f}")
print(f"  nDCG@5 = {sum(ndcgs)/len(ndcgs):.4f}")

# nDCG 는 '순서'를 본다 — 같은 관련 집합, 다른 순서 → recall 같아도 nDCG 다름
print("\n=== nDCG 가 '순서'를 본다: Q1 의 세 가지 순위 ===")
variants = {
    "시스템 순위": ["d2", "d1", "d3", "d4", "d6"],
    "이상적 순위": ["d1", "d2", "d4", "d3", "d6"],   # 등급 높은 순 (3,2,1)
    "나쁜 순위  ": ["d4", "d3", "d6", "d2", "d1"],   # 약한 관련을 위로
}
for name, r in variants.items():
    print(f"  {name}  {r} → R@5={recall_at_k('Q1',r,5):.3f}  nDCG@5={ndcg_at_k('Q1',r,5):.4f}")
print("  → 세 순위 모두 R@5=1.000 이지만 nDCG 는 다르다(순서를 본다).")
```

```
  P@1=0.000  P@3=0.333  P@5=0.400
  R@3=0.500  R@5=1.000
  AP=0.5000   RR=0.5000   nDCG@5=0.6399

=== 컬렉션 전체 (질의 평균) ===
  MAP    = 0.7083
  MRR    = 0.7500
  nDCG@5 = 0.7739

=== nDCG 가 '순서'를 본다: Q1 의 세 가지 순위 ===
  시스템 순위  ['d2', 'd1', 'd3', 'd4', 'd6'] → R@5=1.000  nDCG@5=0.9079
  이상적 순위  ['d1', 'd2', 'd4', 'd3', 'd6'] → R@5=1.000  nDCG@5=1.0000
  나쁜 순위    ['d4', 'd3', 'd6', 'd2', 'd1'] → R@5=1.000  nDCG@5=0.6346
```

### 4.3 결과를 정직하게 읽기 (관찰 포인트)

1. **Q1 vs Q2 — 첫 칸이 운명을 가른다.** Q1 은 1등(d2)이 관련이라 `P@1=1.0, RR=1.0` 으로 출발이 완벽합니다. Q2 는 1등(d6)이 비관련이라 `P@1=0.0, RR=0.5`(첫 정답이 2등) — 두 질의의 *재현율은 둘 다 R@5=1.0 으로 같지만*(상위 5개 안에 정답을 다 건짐), **순위 품질** 은 Q1 이 확연히 높습니다(AP 0.92 vs 0.50, nDCG 0.91 vs 0.64). *집합 지표만 보면 같아 보이는 두 검색이, 순위 지표로는 다르게 평가된다* — 오늘의 핵심.
2. **AP 는 "일찍·빽빽이"를 보상한다.** Q1 의 관련 문서가 1·2·4등에 몰려 AP=0.917 로 높습니다. 만약 4등의 d4 가 더 아래로 밀렸다면 그 지점의 P@i 가 낮아져 AP 가 떨어졌을 겁니다.
3. **nDCG 만이 '순서'를 본다 — recall 이 같아도 다르다.** 마지막 블록이 결정적입니다: Q1 의 세 순위 모두 관련 3개를 top5 안에 다 넣어 **R@5=1.000 으로 동일** 하지만, nDCG@5 는 **1.000(이상적) → 0.908(시스템) → 0.635(나쁨)** 로 갈립니다. 등급 높은 문서(d1=3)를 위로 둘수록 점수가 오릅니다. **"찾았나(recall)" 와 "잘 배치했나(nDCG)" 는 다른 질문** 임을 한눈에 보여줍니다.
4. **왜 단일 지표(MAP)로 요약하나.** 질의마다 P@k 곡선이 다 다른데, 시스템 A vs B 를 *하나의 숫자* 로 비교하려면 MAP·MRR·nDCG 같은 요약이 필요합니다. 다만 요약은 정보를 잃으므로, *목적에 맞는 지표* 를 골라야 합니다(생각거리 2).

## 5. 용어집 (Glossary)

| 용어 | English | 한 줄 뜻 |
|------|---------|---------|
| 관련성 판단 | relevance judgments (qrels) | 질의별로 어떤 문서가 관련인지(또는 등급) 표시한 정답지 |
| 등급 관련성 | graded relevance | 관련을 0/1 이 아닌 단계(예: 0~3)로 매김 |
| Precision@k | precision at k | 상위 k개 중 관련 문서 비율 |
| Recall@k | recall at k | 전체 관련 문서 중 상위 k개로 건진 비율 |
| 평균 정밀도 | Average Precision (AP) | 관련 문서를 만날 때마다의 P@i 평균 |
| MAP | Mean Average Precision | 여러 질의 AP 의 평균 |
| 역순위 | Reciprocal Rank (RR) | 첫 관련 문서 순위의 역수 |
| MRR | Mean Reciprocal Rank | 여러 질의 RR 의 평균 |
| DCG | Discounted Cumulative Gain | 등급을 순위로 할인해 누적한 값 |
| IDCG | Ideal DCG | 이상적 순위의 DCG(최댓값) |
| nDCG | normalized DCG | DCG/IDCG, 0~1 정규화(질의 간 비교 가능) |

## 6. 생각거리 (Reflection)

> [!question] 스스로 답해 보기
> 1. **recall 이 같은데 품질이 다르다?** Q1 과 Q2 는 둘 다 `R@5=1.0` 입니다. 그런데도 두 검색의 *사용자 경험* 이 다르다고 말할 수 있는 이유는 무엇인가요? 어떤 지표가 그 차이를 잡아내나요?
> 2. **어떤 지표를 골라야 하나**: 웹 검색(상위 몇 개가 중요), 질의응답(첫 정답이 중요), 법률 문서 검색(놓치면 안 됨, recall 중요) — 각각에 MRR·nDCG@10·Recall@100 중 무엇이 가장 맞을까요? *지표는 목적을 인코딩한다.*
> 3. **nDCG 의 정규화는 왜 필요한가**: 관련 문서가 1개뿐인 질의와 50개인 질의의 DCG 를 *그냥* 평균하면 무엇이 문제일까요? IDCG 로 나누면(=nDCG) 그게 어떻게 해결되나요?

## 7. 오늘의 한 줄 요약 (Takeaway)

> [!summary]
> 검색의 출력은 *순위 목록* 이므로, [[Day-005]] 의 정밀도·재현율을 **순위를 의식하도록** 확장해 평가한다. **Precision@k·Recall@k** 는 상위 k개의 알짜/누락을, **MAP** 는 관련 문서를 *얼마나 일찍·빽빽이* 끌어올렸는지를, **MRR** 은 *첫 정답이 얼마나 위* 인지를, **nDCG** 는 관련도의 *등급* 과 *위치 할인* 을 함께 본 뒤 이상적 순위(IDCG)로 정규화한다. 핵심 통찰: *같은 문서를 찾아도(=recall 동일) 순서가 다르면 MAP·nDCG 는 달라진다* — "찾았나" 와 "잘 배치했나" 는 다른 질문이다. 직접 구현한 `ir_metrics.py` 에서 Q1(0.91)·Q2(0.64)의 nDCG 차이와, recall 이 같은 세 순위의 nDCG(1.00/0.91/0.63)가 이를 증명한다. 이 지표들이 있어야 [[Day-017]] BM25 의 `k1`·`b` 튜닝과 미니 검색엔진(🛠️ B2) 개선을 *숫자로* 말할 수 있다. 다음은 이 정답지(qrels)를 *현실에서 모으는* 법이다.
