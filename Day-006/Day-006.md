## 0. 들어가며 (From vectors to *words*)

[[Day-003]]–[[Day-005]] 에서 우리는 데이터를 **벡터·행렬**로 다뤘습니다. 그런데 우리가 실제로 검색·요약·대화시키고 싶은 대상은 **글(text)** 입니다. 문제는 — 컴퓨터는 숫자만 안다는 것. 그래서 NLP 의 모든 파이프라인은 **"글 → 숫자"** 로 가는 첫 관문에서 시작합니다.

그 첫 관문이 바로 **토큰화(tokenization)** 입니다. "I love NLP" 라는 문자열을 `["I", "love", "NLP"]` 같은 **의미 단위(token)** 로 쪼개야, 그다음에 셈하고(BoW·TF-IDF, [[Day-007]] 예정), 벡터로 바꾸고(임베딩), 모델에 넣을 수 있습니다.

> Computers only understand numbers, but we want to work with **text**. Every NLP pipeline begins by turning text into pieces — **tokens** — so we can count, vectorize, and feed them to models. Tokenization is that very first gate.

오늘 만든 토크나이저는 단순하지만, **그 위에 Phase 1~6 의 모든 것이 쌓입니다.** ChatGPT 도 여러분이 친 글자를 가장 먼저 토큰으로 쪼갭니다 — 똑같은 일을 오늘 손으로 해 봅니다.

## 1. 핵심 개념 (Core Concepts)

### 1.1 토큰·토큰화·어휘(token · tokenization · vocabulary)

- **토큰(token)**: 텍스트를 처리하기 위해 자른 **최소 단위**. 보통 단어 하나지만, 구두점·서브워드·글자일 수도 있습니다.
- **토큰화(tokenization)**: 원문 문자열을 토큰의 **리스트**로 쪼개는 과정.
- **타입(type) vs 토큰(token)**: 문장 "the cat sat on the mat" 에는 토큰이 6개지만, **서로 다른 단어 종류(type)** 는 `{the, cat, sat, on, mat}` 5개입니다. "the" 는 type 1개·token 2개.
- **어휘(vocabulary)**: 말뭉치(corpus) 전체에 등장한 **서로 다른 토큰의 집합**. 어휘 크기(vocab size)는 모델 입력 차원을 결정하는 중요한 숫자입니다.

> A **token** is the smallest unit we process; **tokenization** splits text into a list of tokens. *Types* are distinct words, *tokens* are their occurrences. The **vocabulary** is the set of distinct tokens in a corpus.

### 1.2 전처리 단계들 (Preprocessing steps)

토큰화 전후로 흔히 거치는 손질입니다. **정답은 없고, 목적에 따라 켜고 끕니다.**

| 단계 | 영어 | 하는 일 | 주의점 |
|------|------|--------|--------|
| 소문자화 | lowercasing | `Apple→apple` 로 통일 | `Apple`(회사)·`apple`(과일) 구분이 사라짐 |
| 정규화 | normalization | 유니코드·이형 통일 (`don’t`→`don't`) | 과하면 의미 손실 |
| 구두점 처리 | punctuation handling | `,`·`.`·`!` 분리 또는 제거 | 감정분석엔 `!`·`?` 가 신호일 수 있음 |
| 불용어 제거 | stopword removal | `the·a·is` 등 흔한 기능어 제거 | 검색엔 유용, 의미 모델엔 해로울 수 있음 |
| 어간추출/표제어 | stemming / lemmatization | `running→run` 으로 통일 | stemming 은 거칠고, lemmatization 은 사전 기반·정확 |

- **어간추출(stemming)**: `studies→studi` 처럼 규칙으로 어미를 잘라냄 — 빠르지만 거침.
- **표제어 추출(lemmatization)**: `studies→study`, `better→good` 처럼 **사전·품사**를 보고 정확한 기본형으로 — 느리지만 정확.

> Preprocessing (lowercasing, normalization, punctuation, stopwords, stemming/lemmatization) is **optional and task-dependent**. Stopword removal helps classic search but can hurt meaning-sensitive models; `!`/`?` may be signal, not noise.

### 1.3 토큰화의 세 가지 단위 (Granularity)

| 단위 | 예: "unhappiness" | 장점 | 단점 |
|------|------------------|------|------|
| **단어(word)** | `["unhappiness"]` | 의미 단위가 명확 | 어휘 폭발·미등록어(OOV) 문제 |
| **글자(character)** | `["u","n","h",...]` | 어휘가 아주 작음·OOV 없음 | 시퀀스가 너무 길고 의미가 옅음 |
| **서브워드(subword)** | `["un","happi","ness"]` | 둘의 절충 — OOV 거의 없음·어휘 적당 | 직관적 경계는 아님 |

- **미등록어 문제(OOV, out-of-vocabulary)**: 학습 때 못 본 단어(`COVID`, 오타, 신조어)를 만나면 단어 토크나이저는 막힙니다.
- **서브워드 토큰화(subword tokenization)** 가 이를 푸는 현대적 답입니다 — 자주 쓰는 조각을 어휘로 두고, 모르는 단어는 **조각으로 분해**합니다. **BPE(Byte-Pair Encoding)**, WordPiece, SentencePiece 가 대표적이며, **GPT·BERT 가 모두 이 방식**입니다. (Phase 4 에서 LLM 토크나이저로 깊이 다룹니다.)

> Word-level tokens hit **OOV** and vocabulary explosion; character-level is OOV-free but too long. **Subword** tokenization (BPE/WordPiece) is the modern compromise behind GPT and BERT — frequent pieces become tokens, unknown words split into known fragments.

### 1.4 한국어는 왜 더 어려운가 (Why Korean is harder)

영어는 **공백(whitespace)** 이 단어 경계를 거의 알려줍니다 (`I love NLP`). 한국어는 다릅니다:

- **교착어(agglutinative)**: 한 어절에 여러 형태소가 붙습니다. "학교에서는" = `학교(명사) + 에서(조사) + 는(보조사)`. 공백으로 자르면 "학교에서는" 통째로 한 토큰이 되어, "학교"·"학교에"·"학교에서"가 전부 **다른 토큰**이 됩니다 → 어휘 폭발.
- 그래서 한국어는 **형태소 분석(morphological analysis)** 이 필요합니다. `KoNLPy`(Okt/Mecab/Kkma 등) 같은 도구가 어절을 형태소로 쪼갭니다.
- 다만 현대 LLM 은 **서브워드(BPE/SentencePiece)** 로 한국어도 통합 처리하는 경우가 많아, 형태소 분석 없이도 동작합니다. (정확도가 중요한 IR·고전 NLP 에선 여전히 형태소 분석이 유용합니다.)

> English whitespace ≈ word boundaries; Korean is **agglutinative**, so one *eojeol* packs several morphemes (`학교+에서+는`). Splitting on spaces explodes the vocabulary, so Korean classically needs **morphological analysis** (KoNLPy), though modern LLMs lean on subword tokenizers.

## 2. 직관 / 비유 (Intuition)

**레고 비유 (The LEGO analogy):**
- 문장은 다 만들어진 **레고 모형**, 토큰은 **블록 한 개**입니다. 토큰화 = 모형을 다시 블록으로 분해하는 일.
- **단어 토큰** = 큰 특수 블록 — 편하지만, 처음 보는 모양(신조어)이면 맞는 블록이 없습니다(OOV).
- **글자 토큰** = 1×1 점 블록만 사용 — 뭐든 만들지만 너무 잘아서 비효율.
- **서브워드 토큰** = 자주 쓰는 중간 크기 블록 세트 — 대부분 이걸로 조립하고, 희귀한 모양만 작은 블록으로 메꿉니다. **그래서 GPT 가 처음 보는 단어도 처리**합니다.

**요리 비유 — 전처리:**
- 토큰화가 재료를 **써는 것**이라면, 전처리는 **씻고 다듬는 것**입니다. 너무 안 다듬으면(원문 그대로) 잡내가 남고(노이즈), 너무 다듬으면(공격적 불용어·stemming) 맛(의미)까지 날아갑니다. **요리(과제)에 맞게** 손질 강도를 정합니다.

> Sentence = a finished LEGO build, tokens = bricks. Word-bricks are convenient but fail on novel shapes (OOV); character-bricks build anything but are tiny; subword-bricks are the practical middle ground GPT uses. Preprocessing is washing/trimming ingredients — too little leaves noise, too much removes flavor (meaning).

## 3. 직관을 굳히는 한 장의 그림 (One picture to remember)

```
원문 (raw text)
   "I'm loving NLP!! 자연어 처리 👍"
        │
        ▼  ① 정규화·소문자화 (normalize / lowercase)
   "i'm loving nlp!! 자연어 처리 👍"
        │
        ▼  ② 토큰화 (tokenize) — 공백·구두점·서브워드 기준
   ["i'm", "loving", "nlp", "!", "!", "자연어", "처리", "👍"]
        │
        ▼  ③ 전처리 (불용어 제거·표제어 등, 선택)
   ["love", "nlp", "자연어", "처리"]
        │
        ▼  ④ 어휘에 매핑 → 정수 ID (다음 편들)
   [42, 7, 311, 312]   →  이제 모델·검색이 다룰 수 있는 '숫자'
```

> Raw text → normalize/lowercase → tokenize → (optional) clean → map to integer IDs. Only after this can models and search engines consume language.

### 4.2 토크나이저 진화시키기 — `tokenizer_basics.py`

가장 단순한 **공백 분리**부터 시작해, **정규식 기반** 으로 진화시키며 차이를 눈으로 봅니다. 아래를 **`tokenizer_basics.py`** 로 저장합니다(옆에 함께 제공).

```python
# tokenizer_basics.py — 공백 분리 → 정규식 토큰화로 진화시키기
import re
import sys
from collections import Counter

sys.stdout.reconfigure(encoding="utf-8")   # Windows 콘솔 한글 깨짐 방지

text = "I'm loving NLP!! Natural-language processing isn't easy, but it's fun."

# (1) 가장 순진한 방법: 공백으로만 자르기
naive = text.split()
print("① split():       ", naive)

# (2) 소문자화 + 정규식: 단어(영문/숫자/아포스트로피)와 구두점을 분리
#     \w+(?:'\w+)? = 단어, 그리고 don't·it's 의 축약형을 한 토큰으로 유지
pattern = re.compile(r"\w+(?:'\w+)?|[^\w\s]")
regex_tokens = pattern.findall(text.lower())
print("② regex 토큰화:  ", regex_tokens)

# (3) 어휘(vocabulary)와 type/token 수 세기
counts = Counter(regex_tokens)
print("토큰 수(tokens):  ", len(regex_tokens))
print("어휘 크기(types): ", len(counts))
print("최빈 토큰 3개:    ", counts.most_common(3))
```

실행:

```powershell
uv run python tokenizer_basics.py
```

기대 출력:

```
① split():        ["I'm", 'loving', 'NLP!!', 'Natural-language', 'processing', "isn't", 'easy,', 'but', "it's", 'fun.']
② regex 토큰화:   ["i'm", 'loving', 'nlp', '!', '!', 'natural', '-', 'language', 'processing', "isn't", 'easy', ',', 'but', "it's", 'fun', '.']
토큰 수(tokens):   16
어휘 크기(types):  15
최빈 토큰 3개:     [('!', 2), ("i'm", 1), ('loving', 1)]
```

**관찰 포인트:**
- `split()` 은 `NLP!!`·`easy,`·`fun.` 처럼 **구두점이 단어에 들러붙어** 따로 셀 수 없습니다. `Natural-language` 도 한 덩어리죠.
- 정규식 버전은 구두점을 **별도 토큰**으로 떼고, `don't`·`it's` 같은 축약형은 **하나로 유지**합니다 — 영어에 흔한 절충안입니다. 다만 `Natural-language` 의 하이픈(`-`)도 **별도 토큰**으로 떨어집니다(`natural` · `-` · `language`). 하이픈을 단어의 일부로 묶을지(`co-operate`) 경계로 볼지(`Natural-language`)는 정답이 없는 **설계 선택**입니다 — 이런 모호함이 토큰화를 까다롭게 만드는 지점입니다.
- type(15) < token(16): `!` 가 두 번 나와 토큰이 하나 더 많습니다. 이 둘의 차이가 [[Day-007]] 의 단어 빈도(BoW·TF-IDF)의 출발점입니다.

### 4.3 전처리 파이프라인 — `preprocess_pipeline.py`

정규화 → 토큰화 → 불용어 제거를 **하나의 파이프라인**으로 묶습니다. 아래를 **`preprocess_pipeline.py`** 로 저장합니다(옆에 함께 제공).

```python
# preprocess_pipeline.py — 정규화 → 토큰화 → 불용어 제거 파이프라인
import re
import sys
import unicodedata

sys.stdout.reconfigure(encoding="utf-8")

# 아주 작은 영어 불용어 목록 (실무에선 nltk/spaCy 의 목록을 씀)
STOPWORDS = {"the", "a", "an", "is", "are", "of", "to", "and", "in", "on", "it"}

def normalize(text: str) -> str:
    # 유니코드 정규화(NFKC): 전각/이형 문자를 표준형으로 통일
    text = unicodedata.normalize("NFKC", text)
    # 곱슬 따옴표 ’ → 곧은 따옴표 '  (don’t → don't)
    text = text.replace("’", "'")
    return text.lower()

def tokenize(text: str) -> list[str]:
    return re.findall(r"\w+(?:'\w+)?|[^\w\s]", text)

def remove_stopwords(tokens: list[str]) -> list[str]:
    # 불용어와 '단독 구두점'을 함께 걸러냄
    return [t for t in tokens if t not in STOPWORDS and not re.fullmatch(r"[^\w\s]", t)]

def preprocess(text: str) -> list[str]:
    return remove_stopwords(tokenize(normalize(text)))

if __name__ == "__main__":
    doc = "The cat is on the mat, and the DOG is in the house!"
    print("원문:        ", doc)
    print("정규화:      ", normalize(doc))
    print("토큰화:      ", tokenize(normalize(doc)))
    print("불용어 제거: ", preprocess(doc))
```

실행:

```powershell
uv run python preprocess_pipeline.py
```

기대 출력:

```
원문:         The cat is on the mat, and the DOG is in the house!
정규화:       the cat is on the mat, and the dog is in the house!
토큰화:       ['the', 'cat', 'is', 'on', 'the', 'mat', ',', 'and', 'the', 'dog', 'is', 'in', 'the', 'house', '!']
불용어 제거:  ['cat', 'mat', 'dog', 'house']
```

**관찰 포인트:**
- 원문 15개 토큰이 불용어·구두점 제거 후 **핵심 4개**(`cat, mat, dog, house`)로 줄었습니다 — 검색 색인에 들어갈 "내용어(content word)"만 남습니다.
- `STOPWORDS` 에서 `is`·`on`·`in` 을 빼 보세요(주석 처리). 감정·문법이 중요한 과제라면 불용어 제거가 **해로울 수도** 있음을 체감할 수 있습니다.
- `normalize` 의 NFKC 가 왜 필요한지 보려면 `doc = "Ｔｈｅ ＣＡＴ"` (전각 문자) 로 바꿔 실행해 보세요.

### 4.4 한국어 토큰화의 함정 — `korean_tokenize.py`

형태소 분석기 없이, **공백 분리만 했을 때 한국어가 왜 무너지는지** 직접 봅니다. 아래를 **`korean_tokenize.py`** 로 저장합니다(옆에 함께 제공).

```python
# korean_tokenize.py — 공백 분리가 한국어에서 왜 어휘 폭발을 일으키나
import sys
from collections import Counter

sys.stdout.reconfigure(encoding="utf-8")

sentences = [
    "학교에 간다",
    "학교에서 공부한다",
    "학교는 멀다",
    "학교를 좋아한다",
]

# 공백으로만 토큰화
ws_tokens = [t for s in sentences for t in s.split()]
print("공백 토큰:    ", ws_tokens)
print("어휘 크기:    ", len(set(ws_tokens)))   # '학교'가 매번 다른 토큰!

# '학교' 가 들어간 어절들 — 전부 서로 다른 토큰으로 셈됨
hak = [t for t in ws_tokens if t.startswith("학교")]
print("'학교' 변형:  ", Counter(hak))
print("→ 사람은 같은 '학교'로 보지만, 공백 토크나이저는", len(set(hak)), "개의 다른 토큰으로 봅니다.")
```

실행:

```powershell
uv run python korean_tokenize.py
```

기대 출력:

```
공백 토큰:     ['학교에', '간다', '학교에서', '공부한다', '학교는', '멀다', '학교를', '좋아한다']
어휘 크기:     8
'학교' 변형:   Counter({'학교에': 1, '학교에서': 1, '학교는': 1, '학교를': 1})
→ 사람은 같은 '학교'로 보지만, 공백 토크나이저는 4 개의 다른 토큰으로 봅니다.
```

**관찰 포인트:**
- `학교에 / 학교에서 / 학교는 / 학교를` — 사람에겐 모두 같은 명사 "학교"지만, 공백 토크나이저에겐 **4개의 전혀 다른 토큰**입니다. 문서가 많아지면 어휘가 걷잡을 수 없이 커집니다(어휘 폭발).
- 형태소 분석기(`KoNLPy`)는 이를 `학교 + 에`, `학교 + 에서` 로 쪼개 "학교"를 **하나의 토큰**으로 모아 줍니다. (`uv add konlpy` 로 설치 가능하나, Java 런타임(JDK)이 필요해 오늘은 개념만 봅니다.)
- 현대 LLM 의 **서브워드** 토크나이저도 비슷하게 "학교"라는 조각을 공유해 이 문제를 완화합니다.

### 4.5 (보너스) 표준 도구로 — `nltk` (선택)

실무에선 검증된 라이브러리를 씁니다. 원리를 이해한 지금, 표준 도구를 만나 봅니다.

```powershell
uv add nltk
```

```python
# nltk_quickstart.py (선택) — 표준 토크나이저와 표제어 추출
import nltk
nltk.download("punkt_tab")          # 문장/단어 토크나이저 데이터
nltk.download("wordnet")            # 표제어(lemmatization) 사전

from nltk.tokenize import word_tokenize
from nltk.stem import WordNetLemmatizer

text = "The cats are running faster than the dogs."
tokens = word_tokenize(text.lower())
lemmatizer = WordNetLemmatizer()
lemmas = [lemmatizer.lemmatize(t) for t in tokens]
print("토큰:   ", tokens)
print("표제어: ", lemmas)   # cats→cat, dogs→dog (running 은 품사정보가 없으면 그대로)
```

```powershell
uv run python nltk_quickstart.py
```

`word_tokenize` 와 `WordNetLemmatizer` — 우리가 4.2–4.3 에서 손으로 만든 일을 검증된 구현으로 대체합니다. **오늘 손으로 이해한 개념이 이 함수들 뒤에 그대로 들어 있습니다.**

## 5. 용어집 (Glossary)

| 용어 | 영어 | 한 줄 뜻 |
|------|------|---------|
| 토큰 | token | 텍스트 처리의 최소 단위(보통 단어·구두점·서브워드) |
| 토큰화 | tokenization | 문자열을 토큰 리스트로 쪼개는 과정 |
| 타입 / 토큰 | type / token | 서로 다른 단어 종류 / 그 등장 횟수 |
| 어휘 | vocabulary | 말뭉치의 서로 다른 토큰 집합 |
| 말뭉치 | corpus | 분석 대상 텍스트 모음 |
| 정규화 | normalization | 유니코드·이형 문자를 표준형으로 통일 |
| 소문자화 | lowercasing | 대소문자를 통일(Apple→apple) |
| 불용어 | stopword | the·a·is 같은 흔한 기능어 |
| 어간추출 | stemming | 규칙으로 어미를 잘라 어근 근사(거침) |
| 표제어 추출 | lemmatization | 사전·품사 기반 정확한 기본형 변환 |
| 미등록어 | OOV (out-of-vocabulary) | 학습 때 못 본 단어 |
| 서브워드 토큰화 | subword tokenization | 단어를 자주 쓰는 조각으로 분해(BPE/WordPiece) |
| BPE | byte-pair encoding | 빈번한 글자쌍을 병합해 어휘를 만드는 서브워드 기법 |
| 교착어 | agglutinative language | 어근에 조사·어미가 붙는 언어(한국어 등) |
| 형태소 분석 | morphological analysis | 어절을 의미 최소단위(형태소)로 분해 |
| 어절 | eojeol | 한국어에서 공백으로 나뉘는 단위 |

## 6. 생각거리 (Reflection)

1. **불용어 제거가 해로운 경우**: "to be or not to be" 라는 문장에서 불용어(`to, be, or, not`)를 모두 제거하면 무엇이 남나요? 불용어 제거가 **검색**엔 유용하지만 **의미·감정 분석**엔 위험할 수 있는 이유를, 이 예로 설명해 보세요. ([[Day-005]] 의 "과제에 따라 도구를 고른다"는 사고와 연결해 보세요.)

## 7. 오늘의 한 줄 요약 (One-line takeaway)

> NLP 의 모든 파이프라인은 **"글 → 토큰 → 숫자"** 로 시작하며, 그 첫 관문이 **토큰화(tokenization)** 다. 공백 분리는 영어에서도 구두점에 막히고, **교착어인 한국어**에선 어휘 폭발로 무너진다. 그래서 **정규화·불용어·표제어** 같은 전처리를 과제에 맞게 켜고 끄며, 현대 LLM 은 OOV 를 푸는 **서브워드(BPE/WordPiece)** 로 글을 쪼갠다. 오늘 손으로 만든 토크나이저 위에 BoW·TF-IDF·임베딩이 차곡차곡 쌓인다.
