## 0. 들어가며 (Motivation)

지금까지의 Phase 2 는 마치 부품을 하나씩 만드는 공정이었습니다. [[Day-016]] 에서 "어떤 단어가 어느 문서에 있나"를 빠르게 찾는 **역색인**을 만들었고, [[Day-017]] 에서 그 후보들에 **BM25** 점수를 매겨 순위를 세웠으며, [[Day-018]] 에서 그 순위가 얼마나 좋은지 **nDCG·MAP** 로 채점하는 법을, [[Day-019]] 에서 그 채점의 **정답지(qrels)** 를 어떻게 만드는지 배웠습니다.

그런데 실제 검색엔진은 이 부품들이 **따로 노는 게 아니라, 정해진 순서로 맞물린 하나의 파이프라인**입니다. 오늘은 그 조립도를 그립니다. 그리고 조립하고 나면 자연히 질문이 생깁니다 — "BM25 의 `k1`, `b` 같은 **손잡이(파라미터)** 는 얼마로 둬야 하지?" 여기서 초보와 실무자가 갈립니다. 초보는 *감으로* 정하고, 실무자는 **지표로 재서** 정합니다. 이 "바꾸고 → 재고 → 더 나은 걸 채택" 하는 루프가 오늘의 두 번째 주제, **평가-주도 개발(EDD)** 입니다.

> The Phase-2 parts (index → rank → evaluate → judgments) aren't separate — a real search engine wires them into one pipeline split across *index-time* and *query-time*. Once assembled, you must choose parameters like BM25's `k1`/`b`. Beginners guess; practitioners **measure**. That measure-change-keep loop is *Evaluation-Driven Development*.

## 1. 핵심 개념 (Core Concepts)

### 1.1 검색 파이프라인의 두 시점 — index-time vs query-time

검색 시스템을 이해하는 가장 중요한 한 문장: **비싼 일은 미리(색인 시점에), 자주 하는 일은 가볍게(질의 시점에).**

| 시점 | 언제 | 하는 일 | 비용 성격 |
|------|------|---------|-----------|
| **색인 시점** (index-time) | 문서가 추가/변경될 때 **한 번** | 토큰화 → 역색인 구축, 문서 길이·`df`·평균 길이(`avgdl`) 등 **통계 저장** | 무겁지만 **드묾** |
| **질의 시점** (query-time) | 사용자가 검색할 때 **매번** | 질의 토큰화 → 역색인 조회 → **BM25 점수화 → 정렬** → 상위 k 반환 | 가벼워야 하고 **잦음** |

핵심 통찰: **모든 문서를 다 훑지 않는다.** 질의 시점에 역색인 덕분에 *질의어를 포함한 문서(후보)만* 점수화합니다([[Day-016]]). 그래서 문서가 수백만이어도 후보는 수천 개로 줄고, 실시간 검색이 가능합니다. BM25 의 `avgdl`·`df` 같은 값도 *질의 때 계산하지 않고* 색인 때 미리 구해 둡니다 — 이것이 속도의 비결입니다.

### 1.2 표준 검색 파이프라인 (retrieval pipeline)

가장 단순한 형태의 파이프라인은 다음과 같습니다:

```
문서 → [토큰화] → [역색인 + 통계]      ……… 색인 시점(1회)
질의 → [토큰화] → [역색인 조회 → BM25 점수 → 정렬 → top-k] → 결과 … 질의 시점(매번)
결과 + qrels → [nDCG/MAP 채점]         ……… 평가(오프라인)
```

실무의 파이프라인은 여기에 **다단계(multi-stage)** 가 붙습니다: 값싼 검색(BM25)으로 **후보를 수백 개로 좁히고(1차 검색, first-stage retrieval)**, 그 위에 무겁지만 정교한 **재순위(re-ranking)** 를 얹는 식(→ Phase 5–6 에서 다룸). 오늘은 그 뼈대인 **1차 검색 + 평가** 까지를 손으로 잇습니다.

### 1.3 평가-주도 개발 (Evaluation-Driven Development)

파이프라인이 생기면 **개선 루프**가 돌 수 있습니다. 개발자가 코드에 TDD(테스트 주도)를 쓰듯, 검색 품질에는 **EDD** 를 씁니다:

1. **베이스라인 측정** — 현재 설정으로 nDCG(등 지표)를 잰다.
2. **하나만 바꾼다** — 예: BM25 `b` 를 0.75 → 0.5. (한 번에 하나여야 원인을 안다.)
3. **다시 측정** — 지표가 올랐나 내렸나?
4. **채택 또는 되돌리기** — 오르면 유지, 내리면 버린다.
5. 반복.

> [!tip] 핵심 규율
> "좋아 보인다"가 아니라 **"지표가 올랐다"** 로 결정한다. 그리고 **한 번에 하나만** 바꾼다 — 두 개를 동시에 바꾸면 무엇이 효과였는지 알 수 없다(과학의 통제 변인과 같다, [[Day-005]]).

### 1.4 손잡이(파라미터)의 뜻 — `k1` 과 `b`

BM25 의 두 파라미터는 [[Day-017]] 에서 배웠습니다. EDD 로 튜닝할 대상이므로 뜻을 다시 새깁니다:

- **`b` (0~1): 문서 길이 정규화 강도.** `b=0` 이면 길이를 아예 무시(긴 문서가 유리 → *키워드 스터핑*에 취약), `b=1` 이면 길이로 강하게 나눔(짧고 밀도 높은 문서 우대). 보통 `0.75`.
- **`k1` (≈1.2~2.0): tf 포화 속도.** 같은 단어가 여러 번 나올 때 점수 증가가 얼마나 빨리 **둔해지는가**. 작을수록 "한 번 나오면 충분" 에 가깝다.

오늘 실습은 `b` 를 EDD 로 튜닝해, **길이 정규화가 실제로 품질(nDCG)을 올리는지**를 숫자로 확인합니다.

## 2. 직관 / 비유 (Intuition)

> [!tip] 비유 — 도서관과 사서
> **색인 시점**은 사서가 밤새 책을 정리해 **카드 목록(=역색인)** 을 만들어 두는 일입니다. 무겁지만 개관 전에 *한 번* 하면 됩니다. **질의 시점**은 낮에 손님이 "요리 책 어디 있어요?" 물을 때 사서가 카드 목록만 보고 *즉시* 안내하는 일 — 서가를 다 뒤지지 않죠(후보만 조회).
>
> **EDD** 는 사서가 "이렇게 배열하니 손님이 더 잘 찾더라"를 **기록으로 확인** 하는 것과 같습니다. 감이 아니라 *대출 통계(=nDCG)* 로 배열을 바꿉니다. "긴 안내문이 무조건 좋다"는 편견(=`b=0`)이 틀렸음을, 통계가 알려 줍니다.
>
> Index-time = the librarian building the card catalog once overnight; query-time = answering each visitor instantly via the catalog (not scanning every shelf). EDD = changing the arrangement based on lending stats (nDCG), not gut feeling.

## 3. 한 장의 그림 (One Picture)

```text
        ┌──────────────── 색인 시점 (INDEX-TIME, 1회) ────────────────┐
문서들 ─▶ 토큰화 ─▶ 역색인(term→postings) + 통계(df, 문서길이, avgdl) 저장
        └──────────────────────────────────────────────────────────┘
                                   │  (미리 계산해 둠)
        ┌──────────────── 질의 시점 (QUERY-TIME, 매번) ──────────────┐
질의 ─▶ 토큰화 ─▶ 역색인 조회(후보만) ─▶ BM25 점수 ─▶ 정렬 ─▶ top-k ─▶ 결과
        └──────────────────────────────────────────────────────────┘
                                   │
결과 + qrels ─▶ nDCG@k 채점 ─▶ ┌─ EDD 루프 ─────────────────────┐
                                 │ b·k1 하나만 바꾼다 → 다시 채점 │
                                 │ 오르면 채택 / 내리면 되돌린다  │
                                 └───────────────▲───────────────┘
                                                 └── 반복 ──┘
```

> Split the engine into a heavy-but-rare index-time stage and a light-but-frequent query-time stage; then close an EDD loop: change one knob, re-score with nDCG, keep only what measurably helps.

### 4.2 조각을 하나로 — `mini_search.py`

아래를 **`mini_search.py`** 로 저장합니다(이 노트 옆에 함께 제공). Phase 2 의 네 조각([[Day-016]]~[[Day-019]])이 그대로 함수 4개가 되어 하나의 파이프라인을 이룹니다: `build_index`(색인) → `bm25_search`(질의) → `ndcg_at_k`(평가) → **EDD 루프**(`b` 탐색).

```python
# mini_search.py — Day-020: 색인 → 랭킹 → 평가를 하나의 파이프라인으로 잇고, 평가-주도로 개선
# 표준 라이브러리(math, re)만 사용. 실행: uv run python mini_search.py
import sys, math, re
sys.stdout.reconfigure(encoding="utf-8")  # Windows 콘솔 한글 깨짐 방지

# 0) 아주 작은 문서 컬렉션 (현실에선 수백만). 검색 주제 + 잡음 문서 섞음.
docs = {
    "d1": "an inverted index maps terms to postings lists for fast search",
    "d2": "bm25 ranks documents using term frequency saturation and length normalization",
    # d3 = 길고 잡다한 블로그: 질의어를 몇 번 흘리지만 대부분 무관한 내용(비관련).
    "d3": ("our marketing blog briefly mentions search engine ranking and search engine "
           "ranking tips but then wanders into gardening travel cooking finance sports "
           "weather music movies books fashion cars phones laptops cameras coffee tea and "
           "many more padding words that make this document quite long overall"),
    "d4": "a search engine uses bm25 ranking to score and order documents",
    "d5": "evaluation metrics like ndcg measure the quality of a ranking",
    "d6": "the cat sat on the mat near the sunny window all afternoon",
    "d7": "tf idf weights each term by its frequency and its rarity in the collection",
    "d8": "a search engine first builds an index then ranks results for a query",
}

def tokenize(text):
    return re.findall(r"[a-z0-9]+", text.lower())

# 1) 색인 파이프라인 (INDEX-TIME) — 한 번만. 역색인 + 문서 길이 + 평균 길이. (Day-016)
def build_index(docs):
    postings = {}       # term -> {doc_id: tf}
    doc_len = {}        # doc_id -> 토큰 수
    for did, text in docs.items():
        toks = tokenize(text)
        doc_len[did] = len(toks)
        tf_local = {}
        for t in toks:
            tf_local[t] = tf_local.get(t, 0) + 1
        for t, f in tf_local.items():
            postings.setdefault(t, {})[did] = f
    N = len(docs)
    avgdl = sum(doc_len.values()) / N
    return {"postings": postings, "doc_len": doc_len, "N": N, "avgdl": avgdl}

# 2) 질의 파이프라인 (QUERY-TIME) — 매번. BM25 로 점수화 후 정렬. (Day-017)
def bm25_search(index, query, k1=1.5, b=0.75, topk=None):
    postings, doc_len, N, avgdl = (index["postings"], index["doc_len"],
                                   index["N"], index["avgdl"])
    scores = {}
    for t in tokenize(query):
        if t not in postings:
            continue
        df = len(postings[t])
        idf = math.log(1 + (N - df + 0.5) / (df + 0.5))   # BM25 IDF (항상 양수)
        for did, f in postings[t].items():
            denom = f + k1 * (1 - b + b * doc_len[did] / avgdl)
            scores[did] = scores.get(did, 0.0) + idf * (f * (k1 + 1)) / denom
    ranked = sorted(scores.items(), key=lambda kv: (-kv[1], kv[0]))  # 점수↓, 동점은 id↑
    return ranked if topk is None else ranked[:topk]

# 3) 평가 (EVAL) — 등급형 nDCG@k. (Day-018)
def dcg(gains):
    return sum(g / math.log2(i + 2) for i, g in enumerate(gains))
def ndcg_at_k(ranked, qrels, k):
    gains = [qrels.get(did, 0) for did, _ in ranked[:k]]
    ideal = sorted(qrels.values(), reverse=True)[:k]
    idcg = dcg(ideal)
    return dcg(gains) / idcg if idcg > 0 else 0.0

# ── 실행 ──
index = build_index(docs)
print("=== (1) 색인 파이프라인 (한 번만 수행) ===")
print(f"  문서 수 N = {index['N']}, 평균 문서길이 avgdl = {index['avgdl']:.2f} 토큰")
print(f"  색인된 고유 단어(term) 수 = {len(index['postings'])}")
print(f"  예: 'search' 의 포스팅(문서→tf) = {index['postings']['search']}")
print(f"      'bm25'   의 포스팅(문서→tf) = {index['postings']['bm25']}")

query = "search engine ranking"
qrels = {"d4": 3, "d8": 3, "d1": 1, "d2": 1, "d5": 1}  # (Day-019) d3·d6·d7 = 0

print(f"\n=== (2) 질의 파이프라인 — query = '{query}' (BM25 기본 k1=1.5, b=0.75) ===")
ranked = bm25_search(index, query)
for rank, (did, sc) in enumerate(ranked, 1):
    g = qrels.get(did, 0)
    mark = "관련" + str(g) if g > 0 else "  ·  "
    print(f"  {rank}. {did}  score={sc:.4f}  [{mark}]  {docs[did][:60]}")

print("\n=== (3) 평가 — nDCG@k (정답지로 채점) ===")
for k in (3, 5):
    print(f"  nDCG@{k} = {ndcg_at_k(ranked, qrels, k):.4f}")

# 4) 평가-주도 개발(EDD) — 길이 정규화 강도 b 를 바꿔가며 nDCG@5 로 최적값 탐색
print("\n=== (4) 평가-주도 개발(EDD): b 를 바꾸면 nDCG@5 는? ===")
best = None
for b in (0.0, 0.25, 0.5, 0.75, 1.0):
    r = bm25_search(index, query, b=b)
    score = ndcg_at_k(r, qrels, 5)
    flag = ""
    if best is None or score > best[1]:
        best = (b, score); flag = "  ← 현재 최고"
    print(f"  b={b:<4}  nDCG@5 = {score:.4f}{flag}")
print(f"  → 이 질의에선 b={best[0]} 가 최적(nDCG@5={best[1]:.4f}). "
      f"'추측' 이 아니라 '측정' 으로 고른다.")
```

```
  b=0.75  nDCG@5 = 0.8562
  b=1.0   nDCG@5 = 0.8562
  → 이 질의에선 b=0.5 가 최적(nDCG@5=0.8562). '추측' 이 아니라 '측정' 으로 고른다.
```

### 4.3 결과를 정직하게 읽기 (관찰 포인트)

1. **색인은 한 번, 조회는 후보만.** (1)블록에서 `'search'` 의 포스팅은 `d1·d3·d4·d8` 뿐입니다 — 질의어를 *포함한 문서만* 후보가 되고, `d6`(고양이)·`d7`(tf-idf) 같은 문서는 애초에 점수 계산조차 하지 않습니다. 이것이 대규모에서 실시간 검색을 가능케 하는 핵심입니다.
2. **길이 정규화가 스팸을 눌러 준다.** `d3` 은 "search engine ranking" 을 흘리지만 대부분 무관한 잡담이라 **비관련(0)** 입니다. 그럼에도 질의어가 여러 번 나와 순위에 끼어듭니다((2)블록 2위). 기본 `b=0.75` 는 이미 `d3` 을 관련 문서 `d4` 아래로 눌렀습니다 — 길이 정규화가 없었다면 더 위로 올라왔을 겁니다.
3. **EDD 가 그걸 숫자로 증명한다.** (4)블록: `b=0`(길이 무시)이면 nDCG@5 가 **0.6780** 인데, `b` 를 0.5 이상으로 올리면 **0.8562** 로 뜁니다. 즉 *길이 정규화를 켜는 것* 만으로 이 질의의 랭킹 품질이 **약 26% 개선**됩니다. "긴 문서를 눌러야 한다"는 직관을, 이제 *지표로* 확인했습니다.
4. **최적값은 측정으로 고른다.** 이 질의에선 `b=0.5` 부터 최고에 도달합니다(그 이상 올려도 이 작은 데이터에선 동일). 실무에선 *여러 질의의 평균* nDCG 로 골라야 한 질의에 과적합되지 않습니다.

### 4.4 (실험) 직접 흔들어 보기

- **`k1` 도 튜닝**: (4)의 루프를 `k1` 값(예: `1.0, 1.5, 2.0`)에 대해서도 돌려, tf 포화 속도가 nDCG 에 주는 영향을 관찰하세요.
- **질의를 늘려 평균 nDCG**: 질의를 2~3개로 만들고 각자 `qrels` 를 준 뒤, EDD 를 *평균 nDCG* 기준으로 바꿔 보세요 — 한 질의 과적합을 피하는 실무 방식입니다.
- **스팸을 더 세게**: `d3` 에 질의어를 더 많이 넣어 보세요. 언제부터 길이 정규화로도 못 누르는지 관찰하면, 왜 실무가 *스팸 탐지·재순위* 같은 추가 단계를 두는지 감이 옵니다.
- **[[Day-019]] pooling 과 연결**: `d3` 같은 문서가 풀에 안 들면 어떻게 평가가 왜곡되는지, 어제 만든 pooling 코드와 이어 생각해 보세요.

## 5. 용어집 (Glossary)

| 용어 | English | 한 줄 뜻 |
|------|---------|---------|
| 검색 파이프라인 | retrieval pipeline | 색인→조회→랭킹→평가로 이어지는 검색 처리 흐름 |
| 색인 시점 | index-time | 문서가 바뀔 때 한 번, 역색인·통계를 미리 만드는 단계 |
| 질의 시점 | query-time | 검색할 때마다 후보를 조회·점수화·정렬하는 단계 |
| 1차 검색 | first-stage retrieval | 값싼 방법(BM25)으로 후보를 좁히는 첫 단계 |
| 재순위 | re-ranking | 좁힌 후보를 정교한 모델로 다시 정렬하는 후속 단계 |
| 평가-주도 개발 | Evaluation-Driven Development (EDD) | 지표로 재서 변경을 채택/기각하는 개선 루프 |
| 길이 정규화 | length normalization | 긴 문서의 tf 이점을 상쇄하는 BM25 의 `b` 항 |
| tf 포화 | tf saturation | 같은 단어 반복의 점수 기여가 둔해지는 성질(`k1`) |
| 평균 문서길이 | avgdl | 컬렉션 평균 토큰 수, BM25 길이 정규화 기준 |

## 6. 생각거리 (Reflection)

> [!question] 스스로 답해 보기
> 1. **왜 `avgdl`·`df` 를 질의 때 계산하지 않고 색인 때 저장할까?** 만약 매 질의마다 다시 계산하면 무엇이 문제가 될까요? (힌트: "자주 하는 일은 가볍게")
> 2. **한 번에 하나만 바꾸는 이유?** EDD 에서 `k1` 과 `b` 를 *동시에* 바꿨더니 nDCG 가 올랐습니다. 무엇을 알 수 없게 될까요?
> 3. **`b=0.5` 가 최적이라는 결론의 함정은?** 단 하나의 질의로 골랐다면 무엇을 과신하는 걸까요? 실무는 이를 어떻게 방지하나요?

## 7. 오늘의 한 줄 요약 (Takeaway)

> [!summary]
> Phase 2 의 네 조각 — 역색인([[Day-016]]) · BM25([[Day-017]]) · nDCG([[Day-018]]) · qrels([[Day-019]]) — 을 하나의 **검색 파이프라인**으로 이었다. 파이프라인은 **색인 시점(무겁지만 1회: 역색인·`avgdl`·`df` 미리 저장)** 과 **질의 시점(가볍고 매번: 후보만 조회→BM25→정렬→top-k)** 으로 갈리며, 후보만 점수화하기에 대규모에서도 실시간이 된다. 조립 뒤엔 파라미터를 **감이 아니라 지표로** 고르는 **평가-주도 개발(EDD)**: *하나만 바꾸고 → nDCG 로 재고 → 오르면 채택* 을 반복한다. 실습에서 길이 정규화 `b` 를 0→0.5 로 켜자 nDCG@5 가 **0.6780→0.8562** 로 올라, "긴 잡담 문서(`d3`)를 눌러야 한다"는 직관을 숫자로 증명했다. 최적값은 *여러 질의 평균* 으로 골라 과적합을 피한다. 이 measure-change-keep 루프가 다음 편 🛠️ **B2 빌드**(from scratch BM25 엔진 + nDCG)의 뼈대다.
