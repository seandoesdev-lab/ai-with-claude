## 0. 들어가며 (Motivation)

Phase 2 의 10편(Day 16–25)을 달려 IR(정보 검색)의 고전을 손에 익혔습니다. [[Day-015]] 에서 "TF-IDF 를 검색 점수로 재활용" 하며 문을 연 뒤, 우리는 **진짜 검색엔진의 뼈대**를 하나씩 세웠습니다 — 후보를 빠르게 좁히는 **역색인**, 점수를 정교하게 매기는 **BM25**, 그 점수가 좋은지 재는 **평가지표**, 이 모두를 잇고 개선하는 **파이프라인·EDD**, 그리고 랭킹을 기하학으로 본 **VSM**.

그런데 [[Day-023]](PRF)와 [[Day-024]](VSM)의 마지막 "생각거리" 에서 우리는 같은 벽에 두 번 부딪혔습니다: **`car` 와 `automobile` 은 뜻이 같은데, 이 모델들은 그걸 모른다.** 오늘은 이 벽을 *어물쩍 넘기지 않고* 정면에서 봅니다. 왜냐하면 이 벽이야말로 — 속도(역색인)나 점수 품질(BM25)의 문제가 아니라 — **"의미를 못 본다"** 는 더 깊은 한계이고, 바로 이것이 딥러닝·임베딩·트랜스포머(Phase 3~5)가 존재하는 이유이기 때문입니다.

> Phase 2 built a real search engine's skeleton — inverted index, BM25, evaluation, pipeline/EDD, VSM. But every one of these is a *lexical* model: it counts literal term overlap between query and document. So they all share one deep wall — **vocabulary mismatch**: `car` scores exactly 0 against a document that only says `automobile`. Today we hit that wall on purpose and see why crossing it requires *learned meaning*, not more clever term counting — the reason Phase 3 (deep learning) exists.

## 1. 핵심 개념 (Core Concepts)

### 1.1 Phase 2 를 한 장으로 — 무엇을 세웠나

| 묶음 | 핵심 질문 | 도구 | 편 |
|------|----------|------|----|
| 구조·색인 | 100만 문서에서 어떻게 *빨리* 후보를 좁히나? | 검색엔진 해부 · **역색인(inverted index)** | [[Day-016]] |
| 랭킹 | 후보를 어떤 *점수* 로 줄 세우나? | **BM25**(tf 포화·길이 정규화), **VSM/코사인** | [[Day-017]], [[Day-024]] |
| 평가 | 그 순위가 *좋은지* 어떻게 아나? | Precision@k·MAP·MRR·**nDCG**, qrels·pooling·κ | [[Day-018]], [[Day-019]] |
| 통합·개선 | 어떻게 *잇고 반복 개선* 하나? | 파이프라인, **평가-주도 개발(EDD)**, 질의 확장(PRF) | [[Day-020]], [[Day-023]] |
| 이론·구현 | *원전* 은? *from scratch* 로? | 📄 확률적 적합성 프레임워크, 🛠️ **B2 빌드** | [[Day-022]], [[Day-021]] |

공통 골격은 늘 **색인 → (질의 처리) → 랭킹 → 평가** 였습니다. 그리고 랭킹의 심장부에는 항상 같은 연산이 있었습니다 — **질의어와 문서어가 얼마나 겹치는가**(BM25 의 $\text{tf}\cdot\text{idf}$, VSM 의 코사인 내적). 이게 강점이자, 오늘 볼 약점의 뿌리입니다.

### 1.2 하나의 뿌리 — 이 모든 게 '어휘(lexical) 매칭'이다

겉으로는 다른 모델처럼 보이지만, Phase 2 의 랭킹은 전부 **같은 가정** 위에 서 있습니다:

$$
\text{score}(q, d) \;=\; \sum_{t \,\in\, q \cap d} (\text{term } t \text{ 의 가중치})
$$

핵심은 **합의 범위가 $q \cap d$** — *질의와 문서에 똑같이 나타난 글자(term)* 뿐이라는 점입니다. BM25 든 VSM 이든, $t$ 가 양쪽에 **글자 그대로** 없으면 그 항은 아예 0 입니다. 그래서:

- `car` 와 `automobile` 은 **교집합이 없어** 기여 0 — 뜻이 같아도 모델에겐 "완전히 남남".
- `car` 와 `cars` 조차 다른 글자라 0 (스테밍 없이는 단/복수도 못 잇는다).
- `Apple`(회사) 과 `apple`(과일) 은 같은 글자라 **똑같이** 매칭 — 다의어(polysemy)를 구분 못 한다.

이것이 **어휘 불일치(vocabulary mismatch)**, 또는 **어휘 간극(lexical gap)** 입니다. IR 의 가장 오래된 골칫거리이자, "단어를 *글자* 로 보는" 모든 모델의 태생적 한계입니다.

### 1.3 Phase 2 가 벽을 완화하려 한 시도 — 그리고 왜 부족했나

우리는 이 벽을 몰랐던 게 아닙니다. Phase 2 안에서도 **우회로**를 두 번 시도했습니다:

1. **질의 확장 / PRF ([[Day-023]])** — 상위 문서에서 단어를 끌어와 질의를 넓혀(`car` → `car vehicle used`) 교집합을 키웠습니다. 하지만 이건 *더 많은 글자를 던지는* 것일 뿐, 여전히 **글자 매칭**입니다. 확장 단어가 우연히 안 겹치면 그만입니다.
2. **스테밍·형태소 정규화** — `cars`→`car`, `running`→`run` 으로 글자를 통일해 일부 간극을 메웁니다. 하지만 `car`↔`automobile` 같은 **동의어**, `bank`(강둑/은행) 같은 **다의어**, 어순·부정(`not good`)은 스테밍으로 절대 못 풉니다.

두 우회로 모두 **"글자를 더 잘 맞추는"** 방향입니다. 근본 문제 — *글자가 아니라 의미를 봐야 한다* — 는 건드리지 못합니다.

### 1.4 벽을 넘는 유일한 길 — '글자'에서 '학습된 의미 벡터'로

[[Day-008]]·[[Day-009]] 에서 이미 씨앗을 봤습니다: **분포 가설(distributional hypothesis)** — "비슷한 문맥에 쓰이는 단어는 비슷한 뜻" — 로 학습한 **단어 임베딩(word embedding)** 은 `car` 와 `automobile` 을 벡터 공간에서 **가깝게** 놓습니다. 그러면 검색 점수를 *글자 교집합* 이 아니라 *벡터의 가까움* 으로 매길 수 있습니다:

$$
\text{score}_{\text{lexical}}(q,d) = \sum_{t \in q \cap d} w_t
\quad\Longrightarrow\quad
\text{score}_{\text{dense}}(q,d) = \cos\big(\text{enc}(q),\ \text{enc}(d)\big)
$$

오른쪽의 `enc(·)` 는 텍스트를 **의미 벡터로 인코딩** 하는 신경망입니다. `enc("car")` 와 `enc("automobile")` 이 가까우면, 글자가 하나도 안 겹쳐도 점수가 높습니다 — **어휘 불일치의 벽이 무너집니다.** 이것이 **밀집 검색(dense retrieval, Phase 5)** 이고, `enc` 를 어떻게 강력하게 만드느냐가 바로 **Phase 3(신경망·트랜스포머)·Phase 4(LLM)** 의 여정입니다.

> [!warning] "그럼 BM25 는 버리나?" — 아니요.
> 밀집 검색이 만능은 아닙니다. 희귀어·고유명사·정확한 코드/ID 검색에선 **글자 매칭(BM25)이 더 정확** 합니다("정확히 이 단어"). 그래서 실무 최강 조합은 **BM25(sparse) + 임베딩(dense)** 을 합치는 **하이브리드 검색**(Phase 5)입니다. Phase 2 는 버리는 게 아니라, 신경망 검색의 **든든한 반쪽**으로 계속 남습니다.

### 1.5 큰 그림에서 지금 위치

```text
Phase 1  텍스트→벡터(TF-IDF·임베딩)·분류      [표현·판단의 기초]
Phase 2  검색엔진: 역색인·BM25·평가·VSM        [★오늘 마무리 — 어휘(lexical) 검색]
   │  └─ 벽: 어휘 불일치 (글자만 봄, 의미 못 봄)
   ▼
Phase 3  신경망·역전파·Attention·Transformer   [의미를 '학습'하는 엔진]
Phase 4  LLM(사전학습·파인튜닝·프롬프팅)         [그 엔진을 언어 전반으로]
Phase 5  임베딩·밀집 검색·하이브리드·리랭킹      [벽을 넘은 검색 = dense retrieval]
Phase 6  RAG (검색 + 생성)                       [검색과 LLM의 결합]
```

오늘로 **Phase 2 가 끝납니다.** 다음 편([[Day-026]])부터 Phase 3 — "글자 매칭"을 "의미 학습"으로 갈아 끼우기 위한 **딥러닝의 기초(신경망·역전파)** 로 들어갑니다.

## 2. 직관 / 비유 (Intuition)

> [!tip] 비유 — 외국어 사전 없이 하는 끝말잇기
> Phase 2 의 검색은 **"똑같은 글자 카드"를 맞추는 게임** 입니다. 손님이 `자동차` 카드를 내밀면, 사서는 서가에서 **글자가 정확히 `자동차` 인 카드** 만 꺼냅니다. 손님이 원한 책이 `자동차` 대신 `승용차`·`차량` 이라 적혀 있으면? — 사서는 **"그런 책 없습니다"** 라고 답합니다. 뜻은 같은데 글자가 달라서요.
> 벽을 넘은 검색(임베딩)은 사서에게 **"뜻 지도"** 를 쥐여 줍니다. `자동차`·`승용차`·`차량` 이 지도 위에서 **바로 옆 동네**임을 알기에, 글자가 달라도 "아, 이걸 찾으시는군요" 하고 꺼내 옵니다. 글자가 아니라 **위치(의미)** 로 찾는 것 — 그 뜻 지도를 *데이터로 학습* 하는 게 Phase 3~5 입니다.
>
> Lexical search is a matching game with identical letter-cards: ask for `car` and you only get cards literally spelling `car` — `automobile` is invisible. Embedding search hands the librarian a *meaning map* where `car`, `automobile`, `vehicle` are neighboring towns, so it fetches by *location (meaning)*, not spelling. Learning that map from data is what Phase 3–5 is about.

## 3. 한 장의 그림 (One Picture)

```text
   질의 q = "car"
      │
      ▼  ┌─────────────── 어휘(lexical) 검색 = Phase 2 ───────────────┐
         │  score(q,d) = Σ  가중치        (t 가 q·d 양쪽에 '글자 그대로')│
         │            t∈ q∩d                                            │
         └──────────────────────────────────────────────────────────────┘
              │  "car" 글자 있음         │  "car" 글자 없음
              ▼                          ▼
        d0 "used car market"       d2 "automobile factory"   ← 뜻은 같은데
           score = 1.57               score = 0.00              점수 0 !!
        d3 "electric cars ..."     d1 "vehicle ..."           (어휘 불일치의 벽)
           score = 0.00 (복수!)       score = 0.00

   ────────────────────────── 벽을 넘으려면 ──────────────────────────
   글자 교집합(q∩d)  ──►  학습된 '의미 벡터'의 가까움
      score = cos( enc(q), enc(d) )       enc = 신경망 인코더
      enc("car") ≈ enc("automobile") ≈ enc("vehicle")  → 이제 다 매칭
                              │
                              ▼
        Phase 3 (신경망·Transformer) → Phase 5 (밀집 검색) → Phase 6 (RAG)
```

> Lexical rankers sum weights only over the literal intersection q∩d, so synonyms and even plurals score 0. Crossing the wall means replacing "term overlap" with the closeness of *learned meaning vectors* — enc(car)≈enc(automobile) — which is the road from Phase 3 (neural nets/Transformers) to Phase 5 (dense retrieval) and Phase 6 (RAG).

### 4.2 BM25 로 '벽' 재현 — `phase2_recap.py` (발췌)

```python
# (발췌) phase2_recap.py — Day-025
# BM25 점수: 질의어가 문서에 '글자 그대로' 없으면 그 항은 0 (핵심!)
def score(self, i, qtoks):
    s = 0.0
    dl = self.len[i]
    for t in qtoks:
        f = self.tf[i].get(t, 0)     # 문서 i 에서 term t 의 빈도
        if f == 0:                   # ← 글자가 없으면 기여 0 (어휘 불일치)
            continue
        denom = f + self.k1 * (1 - self.b + self.b * dl / self.avgdl)
        s += self.idf(t) * (f * (self.k1 + 1)) / denom   # BM25 (Day-017)
    return s
```

```
     자연어의 형태 변화(단/복수)조차 '글자 매칭'은 못 넘는다 (스테밍의 필요).

------------------------------------------------------------------------
(3) 'automobile' 반복(d4)은 BM25 의 tf 포화가 눌러 준다 (Day-017)
------------------------------------------------------------------------
  d4: tf(automobile)=3, |d|=6, score=1.242
  d5: tf(automobile)=1, |d|=10, score=0.636
  d2: tf(automobile)=1, |d|=8, score=0.706
  -> d4 는 3번 반복해도 tf 포화·길이 정규화로 점수가 무한정 오르지 않는다.
     하지만 이건 '같은 글자'의 과잉만 막을 뿐, '다른 글자=같은 뜻'은 못 잇는다.

========================================================================
결론: 역색인·BM25·VSM 은 모두 '질의어와 문서어의 글자 겹침'을 센다.
      -> 동의어·다른 표현은 원천적으로 0점 (어휘 불일치의 벽).
      이 벽을 넘으려면 '글자'가 아니라 '학습된 의미 벡터'가 필요하다.
      -> 신경망·임베딩·트랜스포머(Phase 3) 로 가는 이유.
========================================================================
```

### 4.3 결과를 정직하게 읽기 (관찰 포인트)

1. **(1) 동의어는 0점이 '기본값'이다.** 질의 `automobile` 에 대해, 사실상 같은 주제인 `car`(d0)·`vehicle`(d1)·`cars`(d3) 문서가 **정확히 0.000** 입니다. 점수가 낮은 게 아니라 **아예 후보에도 못 든다** — 역색인이 `automobile` 포스팅 리스트에 이 문서들을 넣지 않기 때문입니다. 이게 어휘 불일치의 냉정한 실체입니다.
2. **(2) 거울상 + 복수형의 벽.** 질의를 `car` 로 바꾸면 이번엔 `automobile` 문서들이 전부 0점으로 뒤집힙니다. 게다가 `d3`("electric **cars**...")는 **뜻이 완전히 같은데도** `cars`≠`car` 라 0점 — **단/복수 형태 변화조차** 글자 매칭은 못 넘습니다(그래서 스테밍이 필요하지만, 스테밍도 동의어는 못 풉니다).
3. **(3) BM25 의 tf 포화는 '다른 문제'를 푼다.** `d4` 는 `automobile` 을 3번 반복(키워드 스터핑)했지만 점수가 3배가 아니라 1.242 에 그칩니다 — [[Day-017]] 의 tf 포화·길이 정규화가 과잉 반복을 눌러 준 겁니다. 이건 **훌륭한 방어**지만, 어디까지나 *같은 글자의 과잉* 만 다스립니다. *다른 글자=같은 뜻* 은 BM25 의 어떤 하이퍼파라미터(k1·b)로도 못 잇습니다 — **차원이 다른 문제**니까요.
4. **정직한 요약.** Phase 2 의 정교함(tf 포화·길이 정규화·nDCG 평가·PRF)은 전부 "글자 매칭이 이미 성립한 뒤" 를 다듬는 기술입니다. **글자가 안 겹치는 그 순간** 은 이 모든 정교함이 **손도 못 대는** 영역 — 그 영역을 여는 열쇠가 Phase 3 의 신경망입니다.

### 4.4 (실험) 직접 흔들어 보기

- **질의를 `vehicle` 로**: `show(bm, "vehicle")` 를 추가해 보세요. d1 만 살고 나머지 자동차 문서는 또 0점 — 벽이 방향과 무관하게 늘 있음을 확인합니다.
- **스테밍 흉내**: `tok()` 에서 끝의 `s` 를 떼는 초간단 스테머(`w[:-1] if w.endswith("s") else w`)를 넣어 `car`↔`cars` 가 이어지는지 보세요. 그리고 그래도 `automobile` 은 여전히 0점임을 확인 — 스테밍의 한계.
- **"가짜 임베딩"으로 벽 넘기 미리보기**: `{"car","automobile","vehicle","cars"}` 를 같은 개념으로 묶는 수동 사전(동의어 확장)을 만들어 질의에 더해 보세요. 점수가 살아나죠? 이 수동 사전을 *데이터로 자동 학습* 하는 게 임베딩입니다(Phase 3~5).
- **평가지표로 손실 재기**: [[Day-018]] 의 nDCG 를 가져와, "자동차 문서 전부가 정답" 인 qrels 로 (1)의 랭킹을 채점해 보세요. 어휘 불일치가 nDCG 를 **얼마나** 깎는지 숫자로 나옵니다.

## 5. 용어집 (Glossary)

| 용어 | English | 한 줄 뜻 |
|------|---------|---------|
| 어휘 불일치 | vocabulary / lexical mismatch | 뜻은 같은데 글자가 달라 매칭이 안 되는 문제(car↔automobile) |
| 어휘 간극 | lexical gap | 질의어와 문서어의 표면형 불일치(어휘 불일치와 동의) |
| 어휘적 검색 | lexical / sparse retrieval | 글자(term) 겹침으로 점수 매기는 검색(BM25·VSM) |
| 밀집 검색 | dense retrieval | 학습된 의미 벡터의 가까움으로 검색(Phase 5) |
| 하이브리드 검색 | hybrid retrieval | 어휘(sparse)+밀집(dense) 점수를 합치는 검색 |
| 다의어 | polysemy | 같은 글자가 여러 뜻(bank=강둑/은행) — 글자 매칭이 못 구분 |
| 스테밍 | stemming | 어형을 어간으로 통일(cars→car) — 형태 변화만 완화 |
| 분포 가설 | distributional hypothesis | "비슷한 문맥에 쓰이면 비슷한 뜻" — 임베딩의 근거([[Day-008]]) |

## 6. 생각거리 (Reflection)

> [!question] 스스로 답해 보기
> 1. **벽의 정체.** (1)과 (2)에서 동의어 문서의 점수가 "낮은" 게 아니라 "정확히 0" 이었습니다. 이 0 이 *역색인의 구조*([[Day-016]])와 어떻게 연결되는지 설명해 보세요. 즉, 왜 이 문서들은 애초에 후보에도 못 들까요?
> 2. **정교함 vs 근본.** BM25 의 tf 포화(k1)·길이 정규화(b)와 nDCG 평가([[Day-018]])는 매우 정교합니다. 그런데도 어휘 불일치를 못 푸는 이유를, "이 기술들이 *언제* 작동하는가" 라는 관점에서 한 문장으로 정리해 보세요.
> 3. **왜 하필 신경망인가.** [[Day-023]] 의 PRF(질의 확장)와 스테밍도 벽을 *완화* 는 했습니다. 그런데 왜 그것으로 충분하지 않고, `enc("car")≈enc("automobile")` 을 *학습* 하는 신경망(Phase 3)이 필요할까요? 두 접근의 근본 차이("글자를 더 맞추기" vs "의미를 배우기")로 설명해 보세요.

## 7. 오늘의 한 줄 요약 (Takeaway)

> [!summary]
> **Phase 2(정보 검색 기초) 마무리**: 우리는 역색인([[Day-016]])·BM25([[Day-017]]·[[Day-022]])·평가지표([[Day-018]]·[[Day-019]])·파이프라인/EDD([[Day-020]]·[[Day-021]])·PRF([[Day-023]])·VSM([[Day-024]])로 **작동하는 어휘 기반 검색엔진**을 세웠다. 그러나 이 모든 모델은 결국 *질의어와 문서어의 글자 교집합 $q\cap d$* 위에서 점수를 매기는 **어휘적(lexical) 검색** 이라, **어휘 불일치(vocabulary mismatch)** 라는 하나의 근본 벽을 공유한다 — 오늘 표준 라이브러리 BM25 로 `car` ↔ `automobile`(동의어)·`car`↔`cars`(복수형)가 **정확히 0점**으로 떨어짐을 수치로 확인했다. BM25 의 tf 포화는 *같은 글자의 과잉* 만 다스릴 뿐, *다른 글자=같은 뜻* 은 어떤 하이퍼파라미터로도 못 잇는다. PRF·스테밍은 "글자를 더 맞추는" 우회일 뿐이며, 벽을 진짜 넘는 길은 점수의 근거를 *글자 교집합* 에서 **학습된 의미 벡터의 가까움**($\cos(\text{enc}(q),\text{enc}(d))$)으로 바꾸는 것 — 이것이 **밀집 검색(Phase 5)** 이고, 강력한 `enc(·)` 를 만드는 여정이 곧 **Phase 3(신경망·트랜스포머)·Phase 4(LLM)** 다. 단, BM25 는 버려지지 않고 정확 매칭에 강한 **하이브리드 검색의 반쪽** 으로 남는다. 다음 편([[Day-026]])부터 그 다리를 건너 딥러닝으로 들어간다.
