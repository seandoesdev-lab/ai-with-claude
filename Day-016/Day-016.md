## 0. 들어가며 (Motivation)

[[Day-015]] 의 마지막 실습 `D @ q` 는 6개 문서 *전부* 와 질의를 내적했습니다. 6개라 순식간이었지만, 웹 검색엔진은 수백억 개의 문서를 다룹니다. 질의 하나가 들어올 때마다 모든 문서를 꺼내 점수를 매긴다면 — 한 번의 검색에 며칠이 걸릴 겁니다.

그런데 현실의 검색은 **수십 밀리초** 만에 끝납니다. 비결은 단순합니다: **대부분의 문서는 애초에 볼 필요가 없다.** `"양자 컴퓨터"` 를 검색했는데 그 단어가 *하나도* 안 들어간 문서는 점수가 0 — 계산할 가치가 없습니다. 그렇다면 **"양자" 라는 단어가 든 문서만 미리 알 수 있다면?** 그 목록만 꺼내 점수를 매기면 됩니다. 이 "미리 만들어 둔 단어→문서 목록" 이 바로 **역색인** 이고, 거의 모든 검색엔진(Lucene·Elasticsearch·구글)의 토대입니다.

> [!tip] 오늘의 한 발
> [[Day-015]] 가 "같은 벡터로 검색이 된다" 를 보였다면, 오늘은 "**어떻게 빠르게** 검색하나" 를 다룹니다. 표현(representation)에서 **시스템(system)** 으로 한 칸 이동합니다.

## 1. 핵심 개념 (Core Concepts)

### 1.1 검색 시스템의 큰 구조 — 두 시간대(time)로 나눠 보기

검색 시스템을 이해하는 가장 좋은 방법은 **언제 일어나는 일인가** 로 나누는 것입니다.

**① 색인 시간(indexing time) — 질의가 오기 *전*, 미리 해 두는 일**

```
문서 수집(crawl/ingest) → 전처리·토큰화(Day-006) → 역색인 구축 → 통계 저장(df, 문서길이…)
```

**② 질의 시간(query time) — 사용자가 검색어를 던진 *후*, 빠르게 해야 하는 일**

```
질의 전처리·토큰화 → 역색인으로 후보 문서 추림(candidate retrieval) → 점수·랭킹(scoring) → 상위 k개 반환
```

핵심 통찰: **무거운 일은 색인 시간에 미리** 해 두고, 질의 시간에는 **가벼운 일만** 합니다. 역색인은 색인 시간의 산물이고, 질의 시간의 속도를 위한 투자입니다.

### 1.2 정방향 색인 vs 역색인 (forward vs inverted)

우리가 [[Day-007]]·[[Day-015]] 에서 만든 문서-단어 행렬은 사실 **정방향 색인(forward index)** 입니다 — "문서 → 그 안의 단어들". 검색에는 이걸 **뒤집어야(invert)** 합니다.

| | 정방향 색인 (forward) | 역색인 (inverted) |
|---|---|---|
| 매핑 | 문서 → [단어들] | 단어 → [문서들] |
| 답하는 질문 | "이 문서엔 무슨 단어가?" | "이 단어는 **어느 문서**에?" |
| 검색에 맞나 | ✗ 모든 문서를 훑어야 | ✓ 질의 단어의 목록만 |

검색의 질문은 항상 "이 *단어* 가 어느 문서에 있나" 이므로, 단어를 키로 두는 역색인이 정답입니다.

### 1.3 역색인의 해부학 — 사전(dictionary)과 포스팅(postings)

역색인은 두 부분으로 이뤄집니다.

- **사전(dictionary / term vocabulary)**: 모든 고유 단어(term)의 목록. 각 단어마다 그 단어를 포함한 문서 수(**문서빈도, df**)와 포스팅 리스트로 가는 포인터를 갖습니다.
- **포스팅 리스트(posting list)**: 한 단어가 등장하는 **문서 ID 목록**. 보통 **문서 ID 오름차순으로 정렬** 해 둡니다(뒤에서 볼 교집합 계산을 위해). 각 항목(posting)에는 문서 ID 외에도 그 문서 안 **출현 빈도(tf)** 나 **위치(position)** 를 함께 담을 수 있습니다.

```text
사전(term)        df   포스팅 리스트(doc_id: tf)
─────────────────────────────────────────────
"brilliant"        2 → [0:1] → [4:1]
"acting"           2 → [0:1] → [1:1]
"boring"           2 → [1:1] → [5:1]
"wonderful"        3 → [0:1] → [2:1] → [4:1]
"amazing"          2 → [2:1] → [3:1]
...
```

`tf`(term frequency)를 함께 저장해 두면 [[Day-017]] 의 BM25 점수를 포스팅을 훑으며 바로 계산할 수 있습니다 — 점수 매기기를 위한 준비물입니다.

### 1.4 불린 검색(Boolean retrieval) — 포스팅 교집합/합집합

가장 단순한 검색은 **점수 없이** "질의 단어를 (모두/하나라도) 포함하는 문서" 를 찾는 것입니다.

- **AND 질의** (`brilliant AND acting`): 두 단어의 포스팅 리스트의 **교집합(intersection)**. 둘 다 정렬돼 있으면 두 포인터를 나란히 전진시키며 $O(n+m)$ 에 교집합을 구합니다(merge).
- **OR 질의** (`boring OR dull`): **합집합(union)**.

이 교집합/합집합이 곧 **후보 추림(candidate retrieval)** 입니다. 전체 N개가 아니라 질의 단어가 든 소수의 문서만 다음 단계(랭킹)로 넘깁니다.

> [!warning] 정렬된 포스팅이 중요한 이유
> 포스팅을 문서 ID로 **정렬** 해 두면 교집합을 한 번의 선형 병합(merge)으로 끝낼 수 있습니다. 정렬돼 있지 않으면 매번 집합 자료구조를 새로 만들어야 하거나 $O(n\times m)$ 으로 느려집니다. "색인 시간에 정렬해 두고, 질의 시간엔 병합만" — 1.1 의 원칙이 여기서 구체화됩니다.

### 1.5 왜 빠른가 — 계산량의 차이

문서 N개, 질의 단어가 든 문서가 평균 $f$개($f \ll N$)라 하면:

- **brute force**([[Day-015]]): 모든 문서와 내적 → $O(N)$.
- **역색인**: 질의 단어의 포스팅(길이 $\approx f$)만 병합·채점 → $O(f)$ 수준.

`the` 처럼 거의 모든 문서에 든 단어는 $f \approx N$ 이라 도움이 안 되지만(그래서 [[Day-007]] 의 idf 가 낮음 — 자연히 점수 기여도 작음), `quantum` 처럼 희소한 단어는 $f$ 가 아주 작아 후보가 극적으로 줄어듭니다. **희소한 단어가 검색을 빠르게도, 정확하게도 만든다** 는 [[Day-015]] 1.4 의 관찰이 속도 측면에서 다시 확인됩니다.

## 2. 직관 / 비유 (Intuition)

> [!tip] 비유 — 책 뒤의 '찾아보기(색인)'
> 두꺼운 전공 책에서 "역전파" 가 나온 곳을 찾고 싶다고 합시다. 1쪽부터 끝까지 넘기며 찾으면(brute force) 하루가 갑니다. 대신 책 **맨 뒤의 '찾아보기(index)'** 를 펴면 — `역전파 ……… 142, 287, 301` — 단어가 곧장 **쪽 번호 목록(posting list)** 으로 연결됩니다. 이 찾아보기가 바로 **역색인** 입니다. 출판사는 책을 *만들 때(색인 시간)* 미리 이 목록을 정리해 두고, 독자는 *찾을 때(질의 시간)* 그 목록만 봅니다. "역전파 **그리고** 경사하강" 이 둘 다 나온 쪽을 찾으려면 두 단어의 쪽 목록에서 **공통으로 등장하는 쪽(교집합)** 만 골라내면 됩니다 — 책 전체를 다시 읽지 않고요.
>
> An inverted index is the "index at the back of a textbook": each term points straight to a sorted list of pages (postings). Built once when the book is made (indexing time), used instantly by the reader (query time); an AND query is just the intersection of two page-lists.

## 3. 한 장의 그림 (One Picture)

```text
  색인 시간(미리)                        질의 시간(빠르게)
 ┌──────────────────────────┐
 │ 문서들(corpus)            │           질의: "brilliant acting"
 │  doc0 "...brilliant       │                    │ 토큰화(Day-006)
 │        acting wonderful"  │                    ▼
 │  doc1 "...boring acting"  │           ["brilliant", "acting"]
 │  ...                      │                    │
 └────────────┬─────────────┘                    │ 각 단어의 포스팅 꺼내기
              │ 토큰화 + 뒤집기(invert)            ▼
              ▼                          brilliant → [0, 4]
 ┌──────────────────────────┐           acting    → [0, 1]
 │ 역색인(inverted index)    │                    │
 │  brilliant → [0, 4]       │   AND(교집합) ─────▶ [0]   ← 후보!
 │  acting    → [0, 1]       │   OR (합집합) ─────▶ [0,1,4]
 │  boring    → [1, 5]       │                    │
 │  wonderful → [0, 2, 4]    │                    ▼  (후보만) 점수·랭킹
 │  ...  (정렬된 posting)    │           score → 상위 k개 반환  → Day-017
 └──────────────────────────┘           (전체 N개가 아니라 후보 f개만!)
```

> Build the inverted index once (indexing time); at query time fetch only the postings of the query terms, intersect/union them to get a small candidate set, then score only those — not all N documents.

### 4.2 역색인을 직접 만들기 — `inverted_index.py`

아래를 **`inverted_index.py`** 로 저장합니다(이 노트 옆에 함께 제공). [[Day-015]] 의 같은 영화 리뷰 6편을 이번엔 **역색인** 으로 뒤집고, 불린 AND/OR 검색으로 후보를 좁혀 봅니다.

```python
# inverted_index.py — Day-016: 역색인(inverted index) from scratch + 불린 검색
# 표준 라이브러리만 사용 (설치 불필요). 실행: uv run python inverted_index.py
import sys
from collections import defaultdict, Counter
sys.stdout.reconfigure(encoding="utf-8")  # Windows 콘솔 한글 깨짐 방지

# 0) Day-015 와 같은 작은 컬렉션(corpus) — 영화 리뷰 6편
corpus = [
    "the movie had brilliant acting and a wonderful story",   # doc0
    "a boring and dull film with terrible acting",            # doc1
    "an amazing soundtrack made this film wonderful",         # doc2
    "the plot was confusing but the visuals were amazing",    # doc3
    "i loved the wonderful music and the brilliant cast",     # doc4
    "a dreadful and boring movie i really disliked",          # doc5
]

def tokenize(text):
    return text.lower().split()   # Day-006 의 최소 토큰화(공백 분리)

# 1) 색인 시간(indexing time): 역색인 구축
#    postings[term] = {doc_id: tf}  →  정렬된 (doc_id, tf) 목록으로 확정
class InvertedIndex:
    def build(self, docs):
        self.N = len(docs)
        self.doc_len = [len(tokenize(d)) for d in docs]       # 문서 길이(뒤의 BM25 준비)
        raw = defaultdict(dict)
        for doc_id, text in enumerate(docs):
            for term, tf in Counter(tokenize(text)).items():
                raw[term][doc_id] = tf
        # 포스팅을 doc_id 오름차순으로 '정렬'해 확정 (교집합 병합을 위해)
        self.postings = {t: sorted(d.items()) for t, d in raw.items()}
        self.df = {t: len(p) for t, p in self.postings.items()}  # 문서빈도
        return self

    def docs_for(self, term):                  # 그 단어가 든 문서 ID 집합(정렬 유지)
        return [doc_id for doc_id, _tf in self.postings.get(term, [])]

# 2) 정렬된 두 리스트의 교집합/합집합 (선형 병합, O(n+m))
def intersect(a, b):
    i = j = 0; out = []
    while i < len(a) and j < len(b):
        if a[i] == b[j]: out.append(a[i]); i += 1; j += 1
        elif a[i] < b[j]: i += 1
        else: j += 1
    return out

def union(a, b):
    i = j = 0; out = []
    while i < len(a) and j < len(b):
        if a[i] == b[j]: out.append(a[i]); i += 1; j += 1
        elif a[i] < b[j]: out.append(a[i]); i += 1
        else: out.append(b[j]); j += 1
    out.extend(a[i:]); out.extend(b[j:])
    return out

# 3) 불린 질의: 질의 단어들의 포스팅을 차례로 AND / OR
def boolean_search(idx, query, mode="AND"):
    terms = tokenize(query)
    lists = [idx.docs_for(t) for t in terms]
    if not lists:
        return []
    if mode == "AND" and any(len(l) == 0 for l in lists):
        # AND 인데 한 단어라도 컬렉션에 없으면 결과 없음
        return []
    combine = intersect if mode == "AND" else union
    result = lists[0]
    for l in lists[1:]:
        result = combine(result, l)
    return result

# ── 실행: 역색인 구축 후 들여다보기 ───────────────────────────
idx = InvertedIndex().build(corpus)
print(f"문서 수 N = {idx.N} | 사전(term) 크기 = {len(idx.postings)}")
print("\n[역색인 일부] term (df) → 포스팅 [doc_id:tf]")
for term in ["brilliant", "acting", "wonderful", "boring", "amazing", "the"]:
    postings = idx.postings.get(term, [])
    pretty = " → ".join(f"[{d}:{tf}]" for d, tf in postings)
    print(f"  {term:<10} (df={idx.df.get(term,0)})  {pretty}")

# ── 불린 검색 데모: 후보가 얼마나 좁혀지는가 ──────────────────
print("\n=== 불린 검색 (후보 추림) ===")
for q, mode in [("brilliant acting", "AND"),
                ("brilliant acting", "OR"),
                ("boring dull", "OR"),
                ("wonderful amazing", "AND")]:
    hits = boolean_search(idx, q, mode)
    print(f"\n[{mode}] {q!r}")
    print(f"  후보 문서: {hits}  (전체 {idx.N}개 중 {len(hits)}개만 채점하면 됨)")
    for i in hits:
        print(f"    - doc{i}: {corpus[i]}")

# ── 속도 직관: brute force 대비 절약량 ───────────────────────
q = "brilliant acting"
cand = boolean_search(idx, q, "AND")
print(f"\n=== 계산량 비교: 질의 {q!r} ===")
print(f"  brute force(Day-015): 문서 {idx.N}개 전부 내적  → {idx.N}회")
print(f"  역색인(오늘):         후보 {len(cand)}개만 채점   → {len(cand)}회")
saved = (1 - len(cand) / idx.N) * 100
print(f"  → 채점 대상 {saved:.0f}% 절약 (문서가 많을수록 격차는 더 벌어진다)")
```

```
    - doc0: the movie had brilliant acting and a wonderful story

[OR] 'brilliant acting'
  후보 문서: [0, 1, 4]  (전체 6개 중 3개만 채점하면 됨)
    - doc0: the movie had brilliant acting and a wonderful story
    - doc1: a boring and dull film with terrible acting
    - doc4: i loved the wonderful music and the brilliant cast

[OR] 'boring dull'
  후보 문서: [1, 5]  (전체 6개 중 2개만 채점하면 됨)
    - doc1: a boring and dull film with terrible acting
    - doc5: a dreadful and boring movie i really disliked

[AND] 'wonderful amazing'
  후보 문서: [2]  (전체 6개 중 1개만 채점하면 됨)
    - doc2: an amazing soundtrack made this film wonderful

=== 계산량 비교: 질의 'brilliant acting' ===
  brute force(Day-015): 문서 6개 전부 내적  → 6회
  역색인(오늘):         후보 1개만 채점   → 1회
  → 채점 대상 83% 절약 (문서가 많을수록 격차는 더 벌어진다)
```

> 사전 크기·정렬 순서는 환경에 따라 표시가 약간 다를 수 있으나 **후보 집합과 절약량은 동일** 합니다.

### 4.3 결과를 정직하게 읽기 (관찰 포인트)

1. **뒤집기가 핵심.** [[Day-015]] 의 문서-단어 행렬(정방향)을 단어 키로 뒤집었을 뿐인데, "단어 → 문서" 질문에 즉답하게 됩니다.
2. **AND는 후보를 극적으로 좁힌다.** `brilliant acting` AND 는 6개 중 **1개** 만 남깁니다. 채점(다음 단계)이 6배 → 1로 줄었습니다.
3. **tf를 함께 저장했다.** 포스팅의 `[doc:tf]` 는 [[Day-017]] BM25 점수를 위한 준비물입니다 — 후보를 추린 뒤 이 tf로 점수를 매깁니다.
4. **정렬 병합의 우아함.** `intersect`/`union` 은 두 정렬 리스트를 한 번씩만 훑습니다($O(n+m)$). 색인 시간에 정렬해 둔 덕분입니다.

### 4.4 (실험) 직접 흔들어 보기

- **흔한 단어의 무력함**: `the` 의 포스팅이 얼마나 긴지 보고, `the movie` 를 AND 검색해 보세요. `the` 는 후보를 거의 못 좁힙니다 — idf 가 낮은 단어가 검색에 무력함을 *속도* 관점에서 재확인.
- **OOV(어휘 밖) 질의**: `boolean_search(idx, "quantum physics", "AND")` 처럼 컬렉션에 없는 단어로 검색하면 후보가 비는 것을 확인하세요.
- **위치 색인(positional index) 맛보기**: 포스팅에 tf 대신 **등장 위치 목록** 을 저장하도록 바꿔, `"brilliant acting"` 을 *구절(phrase)* 로(두 단어가 인접한 문서만) 찾는 확장을 상상해 보세요. (구절 검색의 토대)
- **랭킹으로 잇기**: 후보 `hits` 가 나온 뒤, [[Day-015]] 의 TF-IDF 코사인 점수를 **후보에만** 적용해 정렬해 보세요. "역색인으로 좁히고 → 점수로 줄 세운다" 의 전체 그림이 완성됩니다. ([[Day-017]] 예고)

## 5. 용어집 (Glossary)

| 용어 | English | 한 줄 뜻 |
|------|---------|---------|
| 역색인 | inverted index | 단어→그 단어가 든 문서 목록. 검색의 핵심 자료구조 |
| 정방향 색인 | forward index | 문서→그 안의 단어들 (역색인의 반대) |
| 사전 | dictionary / term vocabulary | 모든 고유 단어 + df + 포스팅 포인터 |
| 포스팅 리스트 | posting list | 한 단어가 등장하는 (정렬된) 문서 ID 목록 |
| 포스팅 | posting | 포스팅 리스트의 한 항목(doc_id, 보통 tf·위치 포함) |
| 문서빈도 | document frequency (df) | 그 단어를 포함한 문서 수 |
| 불린 검색 | Boolean retrieval | 점수 없이 AND/OR/NOT 로 문서 집합을 찾기 |
| 후보 추림 | candidate retrieval | 채점 전, 질의 단어가 든 소수 문서만 추려 내기 |
| 색인 시간 / 질의 시간 | indexing / query time | 미리 해 두는 일 / 검색 때 빠르게 하는 일 |

## 6. 생각거리 (Reflection)

> [!question] 스스로 답해 보기
> 1. **왜 '정렬' 인가**: 포스팅을 문서 ID로 정렬해 두면 교집합이 $O(n+m)$ 입니다. 정렬돼 있지 않다면 교집합 비용은 어떻게 달라질까요? 색인 시간에 정렬하는 한 번의 수고가 질의 시간에 무엇을 사 주는지 설명해 보세요.
> 2. **불린의 한계**: 불린 검색은 "포함/불포함" 만 답하고 *순위* 가 없습니다. `brilliant acting` AND 결과가 100개라면 무엇을 맨 위에 둬야 할까요? — 이 질문이 [[Day-017]] **랭킹·BM25** 의 출발점입니다.
> 3. **저장 비용**: 역색인은 모든 단어의 포스팅을 저장합니다. 문서가 많아지면 포스팅이 길어져 용량이 커집니다. [[Day-015]] 1.5 의 "흔한 단어" 문제와 묶어, 어떤 단어의 포스팅이 가장 길지·어떻게 압축할 수 있을지 상상해 보세요. (포스팅 압축·gap encoding 의 씨앗)

## 7. 오늘의 한 줄 요약 (Takeaway)

> [!summary]
> 검색이 빠른 이유는 **대부분의 문서를 보지 않기** 때문이다. 무거운 일은 **색인 시간** 에 미리 — 문서-단어를 뒤집어 **역색인(단어→정렬된 문서 목록=포스팅 리스트)** 을 만들어 둔다. **질의 시간** 엔 질의 단어의 포스팅만 꺼내 **교집합/합집합(불린 검색)** 으로 후보를 좁히고, 그 소수 후보만 다음 단계(랭킹)로 넘긴다. [[Day-015]] 의 brute force $O(N)$ 가 역색인으로 $O(f)$($f\ll N$)가 되며, 포스팅에 함께 저장한 **tf** 는 [[Day-017]] **BM25** 채점의 준비물이 된다.
