## 0. 들어가며 (Motivation)

지금까지 우리는 검색의 조각들을 하나씩 익혔습니다 — 역색인([[Day-016]]), BM25([[Day-017]]), 평가지표([[Day-018]]), 정답지([[Day-019]]), 그리고 이들을 잇는 파이프라인·EDD([[Day-020]]). 어제 코드(`mini_search.py`)는 "흐름을 보여주는" 목적이라 **함수가 한 줄로 쭉 이어진 절차적 스크립트**였습니다.

오늘은 목적이 다릅니다. 🛠️ **빌드 마일스톤** 의 원칙은 *"동작하는 최소 구현을 내 손으로"* 입니다. 그래서 같은 알고리즘을 이번엔 **모듈(module)** 로 재조직합니다:

- **왜 모듈화하나?** 검색엔진은 색인·랭킹·평가가 **서로 다른 시점·다른 책임**을 가집니다([[Day-020]]의 index-time vs query-time). 이걸 클래스·함수로 나누면 (1) `BM25(k1, b)` 처럼 **파라미터를 바꿔 여러 엔진을 찍어내기** 쉽고, (2) 평가 하니스를 엔진과 분리해 **어떤 엔진이든 같은 잣대로 채점**할 수 있으며, (3) 나중에 재순위·다른 스코어러로 **갈아끼우기** 좋습니다.
- **왜 여러 질의인가?** 어제는 질의 하나로 `b` 를 골랐습니다. 오늘은 질의 3개의 **평균** 으로 `(k1, b)` 를 고릅니다 — 한 질의에 **과적합(overfitting)** 하지 않기 위해서입니다([[Day-005]]의 일반화 정신).

> Yesterday's script *showed* the flow; today's build *solidifies* it into a reusable mini engine — an `InvertedIndex`, a parameterized `BM25` scorer, and an evaluation harness decoupled from the engine — then tunes `(k1, b)` by **mean** nDCG over several queries to avoid overfitting to one.

## 1. 핵심 개념 (Core Concepts)

### 1.1 엔진의 세 모듈과 그 경계 (three modules)

이 빌드의 뼈대는 **책임을 세 덩어리로 나누는 것** 입니다.

| 모듈 | 클래스/함수 | 책임 | 시점 |
|------|-------------|------|------|
| **색인기** | `InvertedIndex` | 문서→토큰→`postings`·`doc_len`·`avgdl`·`N` 저장 | index-time (1회) |
| **스코어러** | `BM25(k1, b)` | 후보 조회 + BM25 점수 + 정렬 → top-k | query-time (매번) |
| **평가 하니스** | `evaluate(engine, queries)` | 엔진 산출 랭킹을 qrels로 채점(nDCG·MAP) | offline |

핵심은 **평가 하니스가 엔진을 '주입(inject)' 받는다**는 점입니다 — `evaluate(engine, queries)`. 엔진이 BM25든, 나중에 다른 스코어러든, **하니스는 그대로** 두고 엔진만 바꿔 끼웁니다. 이 분리 덕에 격자 탐색이 "엔진 여러 개를 같은 잣대로 줄 세우기"로 깔끔해집니다.

### 1.2 평가 하니스 — 등급형 nDCG@k 와 이진 MAP 를 함께

[[Day-018]] 에서 배운 두 지표를 하나의 하니스에 담습니다.

- **nDCG@k** (등급형, graded): `qrels` 의 등급(3/2/1/0)을 그대로 이득(gain)으로 써서, *순위가 좋을수록·상위에 관련 문서가 많을수록* 높음. 로그 할인으로 상위권을 더 중시.
- **MAP** (이진, binary): 등급>0 을 "관련"으로 뭉뚱그려, 관련 문서가 **얼마나 앞쪽에 몰렸는지**(평균 정밀도)를 봄. 등급 정보는 버리지만 **순위 전체**를 반영.

둘을 함께 보는 이유: nDCG 는 "등급까지 반영한 상위권 품질", MAP 는 "관련 문서를 앞으로 끌어왔는가"를 각각 말해 줍니다. 한 지표만 보면 놓치는 면이 있습니다.

> [!warning] 함정 — 지표는 정답지(qrels)만큼만 정확하다
> nDCG·MAP 모두 `qrels` 를 진리로 가정합니다. [[Day-019]] 의 **pooling 편향**을 떠올리세요 — 풀에 안 든 문서는 자동으로 0(비관련)이 됩니다. 우리 실습의 `d3`(스팸성 긴 문서)이 만약 관련이었는데 판정에서 빠졌다면 점수가 왜곡됩니다. **엔진을 튜닝하기 전에 정답지부터 의심**하는 습관이 실무 감각입니다.

### 1.3 평균으로 고르기 — 격자 탐색과 과적합 (grid search & overfitting)

`(k1, b)` 후보들을 격자로 만들어 **모든 조합**을 여러 질의에 대해 채점하고, **평균 nDCG@5** 가 가장 높은 조합을 고릅니다.

```
for k1 in {1.0, 1.5, 2.0}:
  for b in {0.0, 0.25, 0.5, 0.75, 1.0}:
     엔진 = BM25(k1, b)
     점수 = mean_over_queries( nDCG@5(엔진, 질의) )
best = argmax 점수
```

여기서 **"여러 질의 평균"** 이 결정적입니다. 질의 하나로 고르면 그 질의에만 좋은 값을 뽑을 위험(과적합)이 있습니다. 실무에선 여기서 한 걸음 더 나가, 튜닝용(**개발셋, dev set**)과 최종 확인용(**테스트셋, test set**)을 나눠 *튜닝에 쓰지 않은 질의로 최종 검증* 합니다([[Day-005]]의 데이터 분할과 같은 논리).

## 2. 직관 / 비유 (Intuition)

> [!tip] 비유 — 엔진과 계기판을 따로 만든다
> **엔진(BM25 스코어러)** 은 자동차 엔진, **평가 하니스** 는 성능을 재는 **다이나모미터(계기판)** 입니다. 엔진을 튜닝(`k1`·`b`)할 때마다 *같은 계기판* 에 올려 마력(nDCG·MAP)을 잽니다. 계기판을 엔진에 붙여 버리면(=코드를 섞으면) 엔진을 바꿀 때마다 계기판도 다시 만들어야 하죠. 그래서 **분리** 합니다.
>
> 그리고 마력은 **한 번의 주행이 아니라 여러 코스의 평균** 으로 평가합니다 — 직선 코스에서만 빠른 엔진을 "최고"라 부르지 않듯, **여러 질의의 평균** 으로 `(k1,b)` 를 고릅니다.
>
> Build the engine (BM25) and the dyno (eval harness) separately, so any engine can be measured on the same dyno; and rate horsepower by the average over several courses (queries), not a single run.

## 3. 한 장의 그림 (One Picture)

```text
                 ┌───────────────── 빌드된 엔진 (모듈) ─────────────────┐
   docs ─▶ InvertedIndex ──(postings, doc_len, avgdl, N)──▶ BM25(k1,b)
                 └──────────────────────────────────────────┬──────────┘
                                                            │ .search(q) → 랭킹
   queries+qrels ───────────────────────────────────────────┤
                                                            ▼
                              evaluate(engine, queries)  ──▶  평균 nDCG@5, MAP
                                                            │
             ┌──────────── EDD 격자 탐색(grid search) ───────┴────────────┐
             │  for k1 in {…}: for b in {…}:  BM25(k1,b) 를 하니스로 채점  │
             │  → 평균 nDCG@5 최고 (k1,b) 채택 (여러 질의 평균 = 과적합 방지)│
             └────────────────────────────────────────────────────────────┘
```

> A modular engine (index → scorer) feeds one shared harness; a grid search over (k1, b) picks the setting with the best *mean* nDCG@5.

### 4.2 엔진 전체 — `bm25_engine.py`

아래를 **`bm25_engine.py`** 로 저장합니다(이 노트 옆에 함께 제공). 네 개의 관심사(토큰화·색인·스코어링·평가)가 **각자의 이름을 가진 모듈**로 나뉜 것을 눈여겨보세요.

```python
# bm25_engine.py — Day-021 (🛠️ B2 빌드): from-scratch BM25 검색엔진 + nDCG/MAP 평가 하니스
# 표준 라이브러리(math, re)만 사용. 실행: uv run python bm25_engine.py
import sys, math, re
sys.stdout.reconfigure(encoding="utf-8")  # Windows 콘솔 한글 깨짐 방지

# 0) 토큰화 (Day-006)
def tokenize(text):
    return re.findall(r"[a-z0-9]+", text.lower())

# 1) 역색인 (INDEX-TIME, 한 번) — Day-016
class InvertedIndex:
    def __init__(self, docs):
        self.docs = docs
        self.postings = {}   # term -> {doc_id: tf}
        self.doc_len = {}    # doc_id -> 토큰 수
        self._build()
    def _build(self):
        for did, text in self.docs.items():
            toks = tokenize(text)
            self.doc_len[did] = len(toks)
            tf_local = {}
            for t in toks:
                tf_local[t] = tf_local.get(t, 0) + 1
            for t, f in tf_local.items():
                self.postings.setdefault(t, {})[did] = f
        self.N = len(self.docs)
        self.avgdl = sum(self.doc_len.values()) / self.N if self.N else 0.0
    def df(self, term):
        return len(self.postings.get(term, {}))

# 2) BM25 스코어러 (QUERY-TIME, 매번) — Day-017
class BM25:
    def __init__(self, index, k1=1.5, b=0.75):
        self.ix, self.k1, self.b = index, k1, b
    def _idf(self, term):
        df = self.ix.df(term)
        return math.log(1 + (self.ix.N - df + 0.5) / (df + 0.5))  # 항상 양수
    def search(self, query, topk=None):
        ix, k1, b = self.ix, self.k1, self.b
        scores = {}
        for t in tokenize(query):
            if t not in ix.postings:
                continue
            idf = self._idf(t)
            for did, f in ix.postings[t].items():
                denom = f + k1 * (1 - b + b * ix.doc_len[did] / ix.avgdl)
                scores[did] = scores.get(did, 0.0) + idf * (f * (k1 + 1)) / denom
        ranked = sorted(scores.items(), key=lambda kv: (-kv[1], kv[0]))
        return ranked[:topk] if topk else ranked

# 3) 평가 하니스 (EVAL) — Day-018
def dcg(gains):
    return sum(g / math.log2(i + 2) for i, g in enumerate(gains))
def ndcg_at_k(ranked, qrels, k):
    gains = [qrels.get(did, 0) for did, _ in ranked[:k]]
    idcg = dcg(sorted(qrels.values(), reverse=True)[:k])
    return dcg(gains) / idcg if idcg > 0 else 0.0
def average_precision(ranked, qrels):
    rel_total = sum(1 for g in qrels.values() if g > 0)
    if rel_total == 0: return 0.0
    hits, precisions = 0, []
    for i, (did, _) in enumerate(ranked, 1):
        if qrels.get(did, 0) > 0:
            hits += 1; precisions.append(hits / i)
    return sum(precisions) / rel_total
def evaluate(engine, queries, k=5):
    ndcgs, aps = [], []
    for q in queries:
        r = engine.search(q["query"])
        ndcgs.append(ndcg_at_k(r, q["qrels"], k))
        aps.append(average_precision(r, q["qrels"]))
    return {"mean_ndcg": sum(ndcgs)/len(ndcgs), "map": sum(aps)/len(aps),
            "per_query_ndcg": ndcgs}

# 4) EDD 격자 탐색 — Day-020
def grid_search(index, queries, k1_grid, b_grid, k=5):
    results = []
    for k1 in k1_grid:
        for b in b_grid:
            m = evaluate(BM25(index, k1=k1, b=b), queries, k=k)
            results.append((k1, b, m["mean_ndcg"], m["map"]))
    results.sort(key=lambda r: (-r[2], r[0], r[1]))
    return results
```

### 4.3 결과를 정직하게 읽기 (관찰 포인트)

1. **엔진은 만들어졌다.** (2)블록: 첫 질의에서 관련 문서 `d4`(관련3)가 1위, `d8`(관련3)이 3위. 스팸성 긴 문서 `d3`(비관련)이 2위로 끼었지만 — 어제와 같은 현상 — 관련 문서 아래로는 못 밀려났습니다. 길이 정규화가 이미 어느 정도 누르고 있음을 보여 줍니다.
2. **평가 하니스가 여러 질의를 한 번에 채점한다.** (3)블록: 질의 3개의 nDCG@5 가 각각 나오고, 그 **평균(0.8830)** 과 **MAP(0.6700)** 가 요약됩니다. 엔진을 바꿔도 이 하니스는 그대로 씁니다.
3. **격자 탐색은 "평평한 고원(plateau)"을 드러낸다.** (4)블록에서 `b ≥ 0.5` 인 조합은 **모두 평균 nDCG@5 = 0.8830** 으로 동률입니다. 즉 이 작은 데이터·질의에서는 *길이 정규화를 켜는 것(b를 0.25→0.5로)* 이 결정적이고(0.7631→0.8830, **약 +16%**), 그 이상은 `k1`·`b` 를 흔들어도 상위 5개 랭킹의 **순서가 안 바뀝니다**. 큰 코퍼스라면 이 고원이 갈라지며 미세한 최적점이 드러납니다.
4. **"최적 (k1=1.0, b=0.5)"의 정직한 해석.** 동률 중 정렬 규칙(작은 값 우선)이 고른 대표일 뿐, *이 값만 특별히 좋다는 뜻이 아닙니다*. 작은 데이터에서 튜닝 결과를 과신하면 안 된다는 산 교훈입니다 — 그래서 실무는 **더 많은 질의 + dev/test 분리**를 씁니다.

### 4.4 (실험) 직접 흔들어 보기

- **격자를 촘촘하게**: `b_grid` 를 `0.5, 0.6, 0.7, 0.8, 0.9` 로 좁혀 보세요. 이 데이터에선 여전히 평평할 겁니다 — 왜 그런지(랭킹 순서가 안 바뀜) 설명해 보세요.
- **질의·문서를 늘려 고원 깨기**: 관련/비관련 경계가 애매한 문서를 추가하면 `b` 에 따라 순위가 갈리기 시작합니다. 언제 고원이 깨지는지 관찰하세요.
- **dev/test 분리**: 질의를 2:1로 나눠, 앞 2개로 `(k1,b)` 를 고르고 **남은 1개로만** 최종 nDCG 를 재 보세요. 튜닝에 안 쓴 질의의 점수가 진짜 성능에 가깝습니다.
- **MAP vs nDCG 가 엇갈리게**: 등급이 높은 문서를 일부러 하위로 보내는 랭킹을 상상해, 두 지표가 다르게 반응하는 경우를 만들어 보세요.

## 7. 오늘의 한 줄 요약 (Takeaway)

> [!summary]
> 🛠️ **B2 빌드**: Phase 2의 조각들([[Day-016]]~[[Day-020]])을 **모듈화된 미니 검색엔진**으로 from scratch 구현했다. 책임을 셋으로 나눴다 — 색인기 `InvertedIndex`(index-time에 `postings`·`avgdl`·`df` 저장), 파라미터화된 스코어러 `BM25(k1,b)`(query-time에 후보만 점수→정렬), 그리고 **엔진과 분리된** 평가 하니스 `evaluate(engine, queries)`(등급형 **nDCG@5** + 이진 **MAP**). 하니스가 엔진을 주입받으므로 어떤 엔진이든 *같은 잣대*로 채점된다. 이 구조 위에서 `(k1,b)` 를 **격자 탐색**하되 **여러 질의의 평균 nDCG@5** 로 골라 과적합을 피했다. 실습에서 길이 정규화를 켜자(`b` 0.25→0.5) 평균 nDCG가 **0.7631→0.8830(+16%)** 로 뛴 뒤 **고원(plateau)** 에 도달했고 — 작은 데이터에선 그 이상 미세 최적점이 안 갈라진다 — "최적 (k1=1.0, b=0.5)"는 동률의 대표일 뿐이라는 **정직한 한계**까지 확인했다. 실무는 여기에 **더 많은 질의 + dev/test 분리**를 더한다. 이로써 Phase 2의 배움이 *내 손으로 동작하는 엔진*으로 굳었다.
