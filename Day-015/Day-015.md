## 1. 핵심 개념 (Core Concepts)

### 1.1 Phase 1 을 한 장으로 — 무엇을 모았나

| 묶음 | 핵심 질문 | 도구 | 편 |
|------|----------|------|----|
| 표현(Representation) | 텍스트를 어떻게 *숫자* 로? | 토큰화 → BoW/TF-IDF → 임베딩 | [[Day-006]]~[[Day-009]] |
| 판단(Decision) | 그 숫자로 무엇을 *맞히나*? | 분류(로지스틱·NB), 언어모델 | [[Day-010]],[[Day-011]],[[Day-013]] |
| 방법론·구현 | 어떻게 *읽고·만드나*? | 논문 3-pass, from-scratch 빌드 | [[Day-012]],[[Day-014]] |

공통 골격은 항상 **벡터화 → 점수화 → 결정** 이었습니다. 바뀌는 건 "점수를 무엇에 쓰느냐" 뿐입니다.

### 1.2 같은 벡터, 다른 과제 — 분류 vs 검색

[[Day-014]] 의 분류기는 학습된 가중치 $w$ 로 문서 벡터 $x$ 에 **점수** 를 매겼습니다($z=w^\top x+b$). 검색은 그 자리에 **질의 벡터 $q$** 를 넣습니다:

$$\text{분류 점수} = w^\top x \quad\Longleftrightarrow\quad \text{검색 점수} = q^\top d$$

분류의 $w$ 는 *데이터로 학습* 한 "긍정/부정 방향" 이었고, 검색의 $q$ 는 *사용자가 던진 질의* 입니다. 둘 다 "어떤 기준 벡터와 문서 벡터의 내적" 이라는 **같은 연산** 입니다. 텍스트를 벡터로 만든 한 번의 수고가 분류·검색·군집 등 여러 과제로 재사용되는 것 — 이게 벡터 표현의 힘입니다.

### 1.3 벡터 공간 모델(Vector Space Model)과 코사인 유사도

IR 의 고전 뼈대인 **벡터 공간 모델** 은 *질의도 하나의 짧은 문서* 로 보고, 질의·문서를 같은 TF-IDF 공간의 벡터로 둔 뒤 **얼마나 같은 방향을 가리키는가** 로 관련도를 잽니다. 그 척도가 **코사인 유사도**([[Day-004]] 에서 거리 vs 유사도로 만난 그 개념):

$$\text{cos}(q,d)=\frac{q^\top d}{\lVert q\rVert\,\lVert d\rVert}$$

여기에 [[Day-007]]·[[Day-014]] 의 **L2 정규화** 가 다시 빛납니다 — 벡터를 미리 길이 1로 맞춰 두면($\lVert q\rVert=\lVert d\rVert=1$), 코사인이 **그냥 내적 $q^\top d$** 로 줄어듭니다. 그래서 "정규화 → 내적" 한 줄이 곧 검색 점수가 됩니다.

> [!warning] 왜 *코사인* 인가 — 길이가 아니라 *방향*
> 긴 문서는 단어가 많아 벡터가 커지기 쉽습니다. 단순 내적(또는 유클리드 거리)은 "긴 문서일수록 점수↑" 라는 편향을 줍니다. 코사인은 길이를 나눠 버리고 **단어 구성의 방향(=주제)** 만 비교하므로, 문서 길이에 휘둘리지 않습니다.

### 1.4 흔한 단어가 저절로 죽는 이유 (idf 의 재발견)

검색에서 `the`, `a`, `and` 같은 단어로 매칭하면 거의 모든 문서가 걸려 쓸모가 없습니다. TF-IDF 는 이를 **자동으로** 처리합니다 — 이 단어들은 거의 모든 문서에 나와 $\text{df}$ 가 크므로 $\text{idf}=\log\frac{1+N}{1+\text{df}}+1$ 이 1 근처로 깔리고, 점수 기여가 미미해집니다. 별도 불용어 사전 없이도 **희소한 단어가 검색을 주도** 하게 되는 것이죠. (단, 완벽하진 않습니다 — 이 한계와 보정이 [[Day-016]] BM25 의 출발점.)

### 1.5 다음 단계의 두 질문 (Phase 2 예고)

오늘 실습은 **모든 문서와 질의의 내적을 전부 계산** 합니다(brute force). 여기서 IR 의 두 큰 질문이 태어납니다:

1. **속도**: 문서가 100만·1억 개라면 매 질의마다 전부 훑을 수 없다. → **역색인(inverted index)**: "단어 → 그 단어를 포함한 문서 목록" 을 미리 만들어, 질의 단어가 든 문서만 후보로 좁힌다. ([[Day-016]])
2. **점수의 질**: 단순 tf 와 길이정규화는 거칠다. 너무 긴 문서, 너무 잦은 단어를 더 정교하게 다루는 랭킹 함수가 필요하다. → **BM25**(tf 포화 + 문서길이 정규화). ([[Day-016]] 이후)

## 2. 직관 / 비유 (Intuition)

> [!tip] 비유 — 도서관 사서의 머릿속
> Phase 1 에서 우리는 **모든 책(문서)을 같은 양식의 "성분표" 로 요약** 하는 법을 배웠습니다(TF-IDF 벡터: 어떤 단어가 이 책의 특색인가). 이제 **분류** 는 사서가 "이 책은 소설인가 비소설인가?" 처럼 *미리 정한 기준* 으로 책을 나누는 일입니다. **검색** 은 손님이 들고 온 메모(질의)를 *똑같은 성분표 양식* 으로 적어, 서가의 책들 중 **성분표가 가장 닮은 책** 을 꺼내 주는 일이고요. 양식(벡터 공간)은 하나, 사서가 하는 일만 다릅니다. 그리고 손님이 100만 권 중에서 찾아 달라면, 사서는 책을 한 권씩 들춰 보지 않고 **"이 단어가 들어간 책 목록"(역색인)** 부터 펼칩니다.
>
> One vector format ("ingredient label") for every document; classification sorts books by preset criteria, search hands back the books whose label best matches the customer's note (query). Same space — only the librarian's task differs. For a million books, the librarian reaches for a word→books index instead of flipping every page.

### 4.2 TF-IDF 를 '검색 점수' 로 — `tfidf_search.py`

아래를 **`tfidf_search.py`** 로 저장합니다(이 노트 옆에 함께 제공). [[Day-014]] 에서 만든 `TfidfVectorizer` 를 그대로 가져와, 이번엔 *분류* 가 아니라 *질의-문서 코사인 유사도* 로 **랭킹** 합니다.

```python
# tfidf_search.py — Day-015: TF-IDF 를 '검색 점수'로 쓰기 (IR 미리보기)
import sys, math
from collections import Counter
import numpy as np
sys.stdout.reconfigure(encoding="utf-8")  # Windows 콘솔 한글 깨짐 방지

# 0) 작은 문서 컬렉션(corpus) — '영화 리뷰' 6개를 검색 대상 문서로
corpus = [
    "the movie had brilliant acting and a wonderful story",   # doc0
    "a boring and dull film with terrible acting",            # doc1
    "an amazing soundtrack made this film wonderful",         # doc2
    "the plot was confusing but the visuals were amazing",    # doc3
    "i loved the wonderful music and the brilliant cast",     # doc4
    "a dreadful and boring movie i really disliked",          # doc5
]

# 1) TF-IDF 벡터라이저 (Day-007 수식 → Day-014 from scratch 재사용)
class TfidfVectorizer:
    def fit(self, docs):
        self.vocab = sorted({w for d in docs for w in d.split()})
        self.index = {w: i for i, w in enumerate(self.vocab)}
        N = len(docs)
        df = Counter(w for d in docs for w in set(d.split()))   # 문서빈도(df)
        self.idf = np.array([math.log((1 + N) / (1 + df[w])) + 1 for w in self.vocab])
        return self
    def transform(self, docs):
        rows = []
        for d in docs:
            tf = np.zeros(len(self.vocab))
            for w, c in Counter(d.split()).items():
                if w in self.index: tf[self.index[w]] = c       # 어휘 밖 단어(OOV) 무시
            vec = tf * self.idf
            norm = np.linalg.norm(vec)
            rows.append(vec / norm if norm > 0 else vec)         # L2 정규화
        return np.vstack(rows)

# 2) 색인: 컬렉션 전체를 TF-IDF 행렬로 (한 번만 계산)
vec = TfidfVectorizer().fit(corpus)
D = vec.transform(corpus)        # (문서수, 어휘수), 각 행 L2 정규화됨
print(f"어휘 크기 V = {len(vec.vocab)} | 문서 행렬 D = {D.shape} (문서 x 단어)")

# 3) 검색: 질의를 같은 공간으로 → 코사인 유사도(=내적)로 랭킹
def search(query, k=3):
    q = vec.transform([query])[0]      # 질의도 '문서처럼' 같은 공간으로
    scores = D @ q                     # L2정규화 → 내적 = 코사인 유사도
    order = np.argsort(scores)[::-1][:k]
    print(f"\n[질의 Query] {query!r}")
    for rank, i in enumerate(order, 1):
        bar = "#" * int(round(scores[i] * 20))
        print(f"  {rank}. score={scores[i]:.3f} {bar:<20} | doc{i}: {corpus[i]}")

for query in ["wonderful brilliant acting", "boring terrible movie", "amazing music soundtrack"]:
    search(query)
```

### 4.3 결과를 정직하게 읽기 (관찰 포인트)

1. **검색이 '말이 된다'.** `wonderful brilliant acting` 질의는 그 단어들이 든 doc0·doc4 를 위로 올립니다. 우리가 *분류* 에 쓰던 그 벡터·그 정규화로, 이제 *검색* 이 됩니다 — 코드의 차이는 `w` 대신 `q` 를 내적한 것뿐.
2. **흔한 단어는 저절로 조용하다.** `the·a·and` 는 idf 가 낮아 점수에 거의 기여하지 않습니다. 별도 불용어 처리 없이 **희소 단어가 검색을 주도** 하는 idf 효과(1.4)를 숫자로 봅니다.
3. **공유 단어가 적으면 점수도 낮다.** 두 번째 질의의 doc0 은 `boring·terrible·movie` 중 **`movie` 하나만** 공유하고, 그 `movie` 마저 흔해서 idf 가 낮습니다 → 점수 0.182 로 바닥권. 코사인이 *겹치는 방향* 만큼만 점수를 준다는 증거입니다.
4. **전부 훑었다(brute force).** `D @ q` 는 6개 문서 *모두* 와 내적했습니다. 6개라 괜찮지만 100만 개면? — 여기서 [[Day-016]] **역색인** 의 필요가 자연히 드러납니다.

### 4.4 (실험) 직접 흔들어 보기

- **idf 끄기**: `self.idf` 를 전부 1로 바꾸면 순수 카운트(BoW) 검색. 흔한 단어가 순위를 어떻게 흐리는지 비교하세요.
- **길이 편향 보기**: 코퍼스에 아주 긴 문서를 하나 추가하고, **L2 정규화를 끈** 버전(`vec/norm` → `vec`)과 비교해 보세요. 긴 문서가 부당하게 위로 오는지 — 코사인/정규화의 효용을 체감합니다.
- **새 질의**: `"film story"` 처럼 여러 문서에 공통인 단어를 넣어, 동점·근소차가 어떻게 갈리는지 관찰하세요.
- **분류와 잇기**: [[Day-014]] 의 `tfidf_logreg.py` 와 나란히 두고, **같은 `TfidfVectorizer` 가 한쪽은 `w·x`(분류), 한쪽은 `q·d`(검색)** 로 쓰이는 대칭을 직접 확인하세요.

## 5. 용어집 (Glossary)

| 용어 | English | 한 줄 뜻 |
|------|---------|---------|
| 정보 검색 | information retrieval (IR) | 질의에 관련된 문서를 모아 *순위* 매겨 돌려주는 일 |
| 질의 | query | 사용자가 던지는 검색어(짧은 '문서') |
| 벡터 공간 모델 | vector space model | 질의·문서를 같은 벡터로 두고 유사도로 관련도 측정 |
| 코사인 유사도 | cosine similarity | 두 벡터의 *방향* 일치도; L2정규화 시 내적과 같음 |
| 랭킹 | ranking | 관련도 점수 내림차순으로 문서를 정렬 |
| 역색인 | inverted index | 단어→그 단어를 포함한 문서 목록(빠른 후보 추림) |
| BM25 | Okapi BM25 | tf 포화·문서길이 정규화를 넣은 고전 랭킹 함수 |
| brute force | brute-force scan | 모든 문서를 빠짐없이 계산하는(느린) 방식 |

## 6. 생각거리 (Reflection)

> [!question] 스스로 답해 보기
> 1. **하나의 벡터, 여러 과제**: 같은 TF-IDF 벡터가 분류($w^\top x$)·검색($q^\top d$)·군집에 모두 쓰입니다. 그렇다면 [[Day-008]]·[[Day-009]] 의 *임베딩* 도 검색에 쓸 수 있을까요? TF-IDF 검색과 무엇이 다를지 미리 상상해 보세요. (→ Phase 5 '밀집 검색'의 씨앗)
> 2. **TF-IDF 검색의 한계**: `car` 로 검색하면 `automobile` 만 든 문서는 못 찾습니다(어휘 불일치, lexical gap). 또 `not good` 의 부정·어순도 무시합니다([[Day-010]]·[[Day-014]] 와 같은 약점). 이 두 한계를 각각 어떤 단계가 풀어 줄지 로드맵에서 짚어 보세요.
> 3. **속도의 벽**: 문서가 1억 개일 때 `D @ q` 가 왜 불가능한지, 역색인이 "질의 단어가 든 문서만" 후보로 좁히면 계산이 어떻게 줄어드는지 직관적으로 설명해 보세요.

## 7. 오늘의 한 줄 요약 (Takeaway)

> [!summary]
> **Phase 1(NLP 기초) 마무리**: 우리는 *텍스트 → 토큰화 → TF-IDF/임베딩 → 점수 → 판단* 의 골격을 손에 익혔다([[Day-006]]~[[Day-014]]). 그 골격은 *분류* 만의 것이 아니다 — **질의도 하나의 문서로 같은 TF-IDF 공간에 두고 코사인 유사도(=L2정규화 후 내적)로 랭킹** 하면 그대로 **정보 검색(IR)** 이 된다(분류의 $w^\top x$ ↔ 검색의 $q^\top d$). 작은 검색엔진을 돌려 idf 가 흔한 단어를 자동으로 죽이고 희소 단어가 검색을 주도함을 확인했고, "모든 문서를 훑는 brute force" 의 한계에서 **역색인** 과 **BM25** 의 필요가 드러났다 — 이것이 [[Day-016]] 부터 시작하는 **Phase 2(정보 검색 기초)** 의 문이다.
