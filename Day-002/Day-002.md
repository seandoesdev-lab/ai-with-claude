## 0. 들어가며 (Why a clean environment first?)

Day 001 에서 우리는 AI의 큰 지도를 그렸습니다. 오늘은 그 지도를 따라 걸을 **신발을 신는 날**입니다. 화려하진 않지만, 이걸 제대로 하지 않으면 앞으로 매일 작은 고통을 겪게 됩니다.

초보자가 가장 흔히 부딪히는 벽은 알고리즘이 아니라 **환경 문제**입니다.

- "분명 책에선 되는데 내 컴퓨터에선 `ModuleNotFoundError` 가 떠요."
- "어제 되던 코드가 오늘 안 돼요. 뭘 깐 적도 없는데…"
- "이 프로젝트는 NumPy 1.26 이 필요하고, 저 프로젝트는 2.0 이 필요한데 둘 다 깔 수가 없어요."

이 모든 문제의 뿌리는 하나입니다: **여러 프로젝트가 같은 Python 한 곳을 공유**하기 때문입니다. 오늘 배우는 **가상환경(virtual environment)** 이 이 문제를 깔끔하게 해결합니다. 그리고 우리는 그것을 **uv** 라는 현대적인 도구로 다룹니다. 오늘 30분을 투자하면, 앞으로 수십 시간의 삽질을 아낍니다.

> The biggest wall beginners hit is not algorithms — it's the environment. We solve it with **uv**, a single, blazing-fast tool that manages virtual environments, packages, lockfiles, and even Python itself.

## 1. 핵심 개념 (Core Concepts)

### 1.1 Python은 "어디에" 설치되어 있나 — 인터프리터와 site-packages

Python으로 무언가를 한다는 것은, 사실 두 가지가 짝을 이룹니다.

1. **인터프리터(interpreter)**: `python.exe` — 코드를 한 줄씩 읽어 실행하는 프로그램.
2. **site-packages**: 설치한 외부 라이브러리(NumPy 등)가 쌓이는 폴더.

```
python.exe  ──실행──▶  당신의 코드
     │
     └─참조─▶  site-packages/   ← numpy, pandas, ... 가 여기 쌓인다
```

문제는, 컴퓨터에 Python을 그냥 설치하면 이 `site-packages` 가 **전역(global)으로 단 하나**라는 점입니다. 모든 프로젝트가 이 한 폴더를 같이 씁니다.

### 1.2 의존성 지옥 (Dependency Hell)

프로젝트 A는 `numpy==1.26`, 프로젝트 B는 `numpy==2.1` 을 원한다고 합시다. 전역 site-packages 는 하나뿐이라, 한 버전을 깔면 다른 프로젝트가 깨집니다. 라이브러리들이 서로의 버전을 두고 충돌하는 이 상황을 **의존성 지옥(dependency hell)** 이라고 부릅니다.

| | 전역 환경 하나만 쓸 때 | 프로젝트마다 가상환경을 쓸 때 |
|---|---|---|
| 버전 충돌 | 자주 발생 😵 | 격리되어 발생 안 함 ✅ |
| 재현성 | "내 컴퓨터에선 됐는데…" | 어디서든 동일하게 재현 ✅ |
| 정리 | 뭘 깔았는지 뒤죽박죽 | 폴더 하나 지우면 깨끗이 삭제 ✅ |

### 1.3 가상환경(Virtual Environment) — 프로젝트 전용 작은 Python

**가상환경**은 프로젝트 폴더 안에, **그 프로젝트만을 위한 작은 Python 환경과 전용 site-packages** 를 만드는 것입니다. 디렉토리 이름은 보통 `.venv` 입니다.

```
my-project/
├─ .venv/                  ← 가상환경 (이 프로젝트 전용)
│   ├─ Scripts/            ← python.exe 등 (Windows)
│   └─ Lib/site-packages/  ← 이 프로젝트 전용 라이브러리만 여기 쌓임
├─ pyproject.toml          ← 이 프로젝트가 필요로 하는 패키지 "선언"
└─ uv.lock                 ← 실제로 설치된 정확한 버전 "잠금"
```

- 프로젝트가 끝나면? `.venv` 폴더만 지우면 흔적 없이 사라집니다. 전역 Python은 손대지 않았으니 안전합니다.

> A virtual environment is a per-project mini-Python with its own `site-packages`. Delete the folder and it's gone — the global Python stays clean.

### 1.4 uv — 환경·패키지·Python을 하나로 다루는 초고속 도구

전통적으로는 이 일을 여러 도구로 나눠 했습니다: 가상환경은 `venv`, 설치는 `pip`, 버전 고정은 `pip-tools`, Python 버전 관리는 `pyenv`… 도구가 많고 느렸습니다.

**uv** (Astral 사가 Rust로 만든 도구)는 이 모든 것을 **하나의 빠른 명령어**로 통합합니다.

| 하고 싶은 일 | 전통 방식 | uv 방식 |
|---|---|---|
| 가상환경 만들기 | `python -m venv .venv` | `uv venv` (또는 `uv init` 가 자동 생성) |
| 패키지 설치 | `pip install numpy` | `uv add numpy` |
| 코드 실행 | 활성화 후 `python x.py` | `uv run python x.py` (활성화 불필요) |
| 버전 고정/복원 | `pip freeze` / `pip install -r` | `uv.lock` 자동 / `uv sync` |
| Python 자체 설치 | 수동 다운로드 | `uv python install 3.12` |

핵심 장점:
- **빠르다**: pip 대비 수십 배 빠른 설치 (Rust + 강력한 캐시).
- **활성화가 필요 없다**: `uv run` 이 알아서 그 프로젝트의 `.venv` 안에서 실행합니다. → Windows에서 `Activate.ps1` 실행 정책 오류로 고생할 일이 줄어듭니다.
- **재현성이 기본**: `pyproject.toml`(선언) + `uv.lock`(정확한 버전 잠금)을 자동으로 관리합니다.
- **pip 명령도 그대로**: 기존 습관이 있다면 `uv pip install ...` 처럼 pip 호환 명령도 됩니다.

### 1.5 `pyproject.toml` 과 `uv.lock` — 무엇을 설치했는지 기록하기

- **`pyproject.toml`**: "이 프로젝트는 numpy, jupyterlab 이 필요하다"는 **사람이 읽는 선언**. `uv add` 를 하면 여기에 자동으로 적힙니다.
- **`uv.lock`**: 그 선언을 만족하는 **정확한 버전들(예: numpy 2.1.3)과 해시를 기계가 잠가 둔 파일**. 어떤 컴퓨터에서든 똑같이 복원됩니다.

다른 컴퓨터(또는 미래의 나)는 이 두 파일만 있으면 `uv sync` 한 줄로 **완전히 동일한 환경**을 복원합니다. 이것이 **재현성(reproducibility)** 의 핵심입니다 — 연구·실무에서 "내 환경에서만 되는 코드"는 가치가 절반으로 깎입니다.

### 1.6 Jupyter — 한 칸씩 실행하며 "대화하는" 환경

AI/데이터 작업은 **탐색적(exploratory)** 입니다. 데이터를 보고, 숫자를 출력하고, 그래프를 그리고, 다시 고치고… 이 반복에 일반 스크립트(.py)는 불편합니다. 전체를 매번 처음부터 다시 돌려야 하니까요.

**Jupyter Notebook / JupyterLab** 은 코드를 **셀(cell)** 단위로 쪼개, 셀 하나씩 실행하고 결과를 바로 아래에서 보는 환경입니다. 변수는 메모리에 남아 있으므로, 무거운 데이터를 한 번만 불러두고 그 위에서 계속 실험할 수 있습니다. 이번 트랙의 실습 상당수가 Jupyter 에서 진행됩니다.

### 1.7 NumPy — 왜 첫 라이브러리가 NumPy 인가

NumPy(Numerical Python)는 **고속 수치 배열(array)** 라이브러리입니다. 앞으로 배울 거의 모든 것 — 임베딩 벡터, 행렬 연산, PyTorch 텐서 — 이 NumPy 의 사고방식 위에 서 있습니다. 핵심은 **벡터화(vectorization)**: 숫자 100만 개에 같은 연산을 할 때, `for` 반복문 대신 배열 한 줄로 처리하면 훨씬 빠르고 간결합니다. 오늘은 그 첫 한 줄을 맛봅니다.

## 2. 직관 / 비유 (Intuition)

**주방 비유 (The Kitchen Analogy):**

- 전역 Python 하나만 쓰는 것은, **온 가족이 같은 도마·칼·재료통 하나**를 공유하는 것과 같습니다. 누군가 설탕통에 소금을 넣어두면, 다른 사람의 요리가 망합니다.
- **가상환경**은 **요리(프로젝트)마다 전용 조리대를 따로 차리는 것**입니다. 김치찌개용 조리대와 베이킹용 조리대가 분리되어 있으니 서로 망칠 일이 없습니다.
- **uv** 는 그 조리대를 **순식간에 차려 주는 만능 셰프 도구**입니다 — 조리대 설치(venv), 재료 주문(install), 레시피 기록(lock)을 한 손으로 처리합니다.
- **`pyproject.toml` + `uv.lock`** 은 그 요리의 **레시피와 정확한 재료 분량표**입니다. 이 둘만 있으면 다른 주방에서도 똑같은 요리를 재현할 수 있습니다.
- **Jupyter** 는 **맛보면서 간을 맞추는 요리법**입니다. 한 솥 다 끓이고 나서 맛보는 게 아니라, 한 단계씩 넣고 맛보고 조정합니다.

> Global Python = one shared cutting board for the whole family. A virtual environment = a separate prep station per dish. **uv** = the all-in-one chef's tool that sets up the station, orders ingredients, and records the recipe — fast. `pyproject.toml` + `uv.lock` = the recipe + exact amounts, so any kitchen reproduces the dish.

## 3. 실습 (Hands-on)

> 환경: **Windows + PowerShell**. 아래 명령은 PowerShell 창에 그대로 입력하면 됩니다.

### 3.1 uv 설치하기

PowerShell을 열고, 공식 설치 스크립트를 실행합니다 (관리자 권한 불필요, 현재 사용자에게 설치):

```powershell
powershell -ExecutionPolicy ByPass -c "irm https://astral.sh/uv/install.ps1 | iex"
```

설치가 끝나면 **PowerShell 창을 닫고 새로 엽니다** (PATH 갱신을 위해). 그다음 확인:

```powershell
uv --version
```

`uv 0.x.x` 같은 줄이 나오면 성공입니다.

> 💡 다른 설치 방법도 있습니다: 이미 `pip` 가 있다면 `pip install uv`, 또는 `winget install astral-sh.uv`. 무엇으로 깔든 결과는 같습니다.

### 3.2 (필요 시) Python 자체도 uv 로

uv 는 Python 인터프리터까지 관리할 수 있습니다. 시스템에 Python 이 없거나 버전을 맞추고 싶다면:

```powershell
uv python install 3.12
```

uv 가 필요할 때 알아서 적절한 Python 을 받아 쓰므로, 사실 이 단계는 건너뛰어도 다음 단계에서 자동 처리됩니다.

### 3.3 프로젝트 만들기 (`uv init`)

오늘 실습용 프로젝트를 만듭니다. (원하는 위치로 바꿔도 됩니다.)

```powershell
cd $HOME
uv init ai-study-day002
cd ai-study-day002
```

폴더 안을 보면(선택) uv 가 `pyproject.toml`, `README.md`, `main.py`, `.python-version` 등을 자동으로 만들어 둔 것을 볼 수 있습니다:

```powershell
ls
```

`pyproject.toml` 이 이 프로젝트의 **설계도**입니다. 아직 가상환경(`.venv`)은 보이지 않을 수 있는데, 다음 단계에서 패키지를 추가하는 순간 uv 가 자동으로 만들어 줍니다.

### 3.4 패키지 추가하기 (`uv add`) — 가상환경은 자동 생성

이제 오늘 필요한 라이브러리를 추가합니다 — **NumPy** 와 **JupyterLab**:

```powershell
uv add numpy jupyterlab
```

이 한 줄이 일어나는 일:
1. `.venv` 가상환경을 (없으면) **자동으로 생성**하고,
2. numpy·jupyterlab 과 그 의존성들을 **그 안에** 초고속으로 설치하며,
3. `pyproject.toml` 에 의존성을 적고, `uv.lock` 에 **정확한 버전**을 잠가 둡니다.

무엇이 깔렸는지 확인:

```powershell
uv pip list
```

`numpy`, `jupyterlab`, 그리고 그들이 끌고 온 의존 패키지들이 보입니다. 이 모든 게 **전역이 아니라 이 프로젝트의 `.venv` 안에만** 설치되었다는 점이 핵심입니다.

> 📌 **활성화(activate)를 안 해도 됩니다.** 전통적 방식에서는 `.\.venv\Scripts\Activate.ps1` 로 가상환경을 켜야 했고, 여기서 Windows 실행 정책 오류가 자주 났습니다. uv 는 다음 단계의 `uv run` 으로 이 과정을 통째로 건너뜁니다.

### 3.5 첫 NumPy 한 줄 — 벡터화 체험 (`uv run`)

아래 내용을 프로젝트 폴더에 **`first_numpy.py`** 로 저장합니다. (이 파일은 옆에 함께 제공됩니다.)

```python
# first_numpy.py — 첫 NumPy 배열과 "벡터화" 체험
import numpy as np

# 1) 파이썬 리스트가 아니라 NumPy 배열(ndarray)을 만든다
a = np.array([1, 2, 3, 4, 5])
print("배열 a       :", a)

# 2) 벡터화(vectorization): 반복문 없이 '한 줄'로 모든 원소에 연산
print("a * 2        :", a * 2)          # [ 2  4  6  8 10]
print("a 의 제곱     :", a ** 2)         # [ 1  4  9 16 25]
print("a 의 평균     :", a.mean())       # 3.0
print("a 의 표준편차 :", a.std().round(3))

# 3) 2차원 배열(행렬)과 모양(shape)
M = np.array([[1, 2, 3],
              [4, 5, 6]])
print("행렬 M 의 shape:", M.shape)       # (2, 3) → 2행 3열
print("M 의 전체 합   :", M.sum())        # 21
print("열별 합(axis=0):", M.sum(axis=0)) # [5 7 9]
```

실행 — **활성화 없이** uv 가 프로젝트의 `.venv` 안에서 돌려 줍니다:

```powershell
uv run python first_numpy.py
```

**관찰 포인트:**
- `a * 2` 가 원소 하나하나에 자동 적용됩니다. 파이썬 리스트였다면 `[x*2 for x in a]` 처럼 반복문이 필요했습니다. 이것이 **벡터화** 입니다.
- `.shape`, `.mean()`, `.sum(axis=0)` 같은 속성·메서드가 곧 임베딩·행렬 연산에서 매일 쓰는 도구가 됩니다.
- `uv run` 이 `numpy` 를 어디서 찾아 실행했는지 생각해 보세요 — 전역이 아니라 이 프로젝트의 `.venv` 입니다.

### 3.6 (보너스) 벡터화가 정말 빠른가 — 직접 측정

NumPy 가 단지 간결한 게 아니라 **빠르다**는 걸 숫자로 확인해 봅시다. 아래를 `speed_test.py` 로 저장하고 실행:

```python
# speed_test.py — 순수 파이썬 반복 vs NumPy 벡터화 속도 비교
import time
import numpy as np

N = 5_000_000  # 500만 개

# (A) 순수 파이썬 리스트 + 반복문
py_list = list(range(N))
t0 = time.perf_counter()
py_result = [x * 2 for x in py_list]
t1 = time.perf_counter()
print(f"순수 파이썬 반복 : {t1 - t0:.4f} 초")

# (B) NumPy 배열 + 벡터화
np_arr = np.arange(N)
t2 = time.perf_counter()
np_result = np_arr * 2
t3 = time.perf_counter()
print(f"NumPy 벡터화     : {t3 - t2:.4f} 초")

print(f"NumPy 가 약 {(t1 - t0) / (t3 - t2):.1f} 배 빠름")
```

```powershell
uv run python speed_test.py
```

대개 NumPy 가 **수십 배** 빠르게 나옵니다. 이유는 NumPy 내부 연산이 C로 구현되어 있고, 데이터가 메모리에 촘촘히 붙어 있기 때문입니다. (자세한 원리는 Phase 0 후반에 다룹니다.)

### 3.7 환경을 기록하고 재현하기 — `uv.lock` 과 `uv sync`

좋은 소식: **이미 끝났습니다.** `uv add` 를 한 순간 `pyproject.toml` 과 `uv.lock` 이 자동으로 갱신되었습니다. 열어서 확인해 보세요(선택):

```powershell
cat pyproject.toml
```

`dependencies = ["numpy>=...", "jupyterlab>=..."]` 처럼 의존성이 선언되어 있고, `uv.lock` 에는 **정확한 버전과 해시**가 잠겨 있습니다.

다른 컴퓨터(또는 미래의 나)는 이 프로젝트를 받은 뒤 단 한 줄로 **완전히 동일한 환경**을 복원합니다:

```powershell
uv sync
```

`uv sync` 는 `uv.lock` 을 그대로 재현합니다 — 같은 버전, 같은 결과. 이것이 **재현 가능한 환경**의 핵심입니다.

> 🔁 pip 세계와 주고받아야 한다면, `uv export --format requirements-txt > requirements.txt` 로 전통적인 `requirements.txt` 도 뽑아낼 수 있습니다.

### 3.8 Jupyter 열어보기 (선택, 강력 추천)

같은 폴더에서 — 역시 활성화 없이:

```powershell
uv run jupyter lab
```

브라우저가 열리며 JupyterLab 이 뜹니다. 왼쪽에서 새 노트북(Notebook)을 만들고, 셀에 아래를 한 줄씩 입력하며 `Shift+Enter` 로 실행해 보세요:

```python
import numpy as np
np.array([1, 2, 3]).mean()
```

결과 `2.0` 이 셀 바로 아래 출력됩니다. 셀을 나눠 실험하는 이 감각이, 앞으로의 데이터 탐색에서 큰 무기가 됩니다. (종료는 PowerShell 창에서 `Ctrl + C` 두 번)

### 3.9 (참고) pip 습관이 있다면 — uv 의 pip 호환 모드

기존에 `pip`/`venv` 에 익숙하다면, uv 는 그 명령들을 **더 빠르게** 흉내 내는 모드도 제공합니다:

```powershell
uv venv                      # .venv 생성 (python -m venv .venv 대체)
uv pip install numpy         # pip install 대체 (활성화돼 있으면 그 venv에)
uv pip freeze                # pip freeze 대체
```

`uv add`/`uv run`/`uv sync` 의 **프로젝트 방식**이 더 권장되지만(잠금파일을 자동 관리), 빠르게 pip 처럼 쓰고 싶을 때 이 모드가 유용합니다.

## 5. 생각거리 (Reflection)

스스로 답해 보세요 (정답보다 사고 과정이 목적):

1. 친구가 "나는 그냥 전역에 다 깔고 쓰는데 잘만 돼"라고 합니다. 프로젝트가 늘어나면 **어떤 문제**가 생길지, 1.2의 표를 근거로 설명해 보세요.
	- 전역 환경에서 사용하는 경우, 버전 관리가 어렵고, 재현성으로 다른 환경에서는 동일하게 작동되지 않는 이슈가 발생할 수 있으며,  특정 프로젝트에서 어떤 라이브러리가 필요한 지 파악이 어려움.
2. uv 에서 `pyproject.toml` 과 `uv.lock` 은 역할이 어떻게 다를까요? "사람이 원하는 것"과 "기계가 실제로 깐 것"이라는 관점에서, 왜 **둘 다** 필요한지 설명해 보세요. (힌트: `numpy>=1.26` vs `numpy==2.1.3`)
	- pyproject.toml에는 의존성들이 명시 되어 있고, uv.lock에는 해당 의존성에 대한 버전와 해시가 명시 되어 있음.
3. uv 의 `uv run` 은 가상환경을 **활성화하지 않고도** 코드를 그 환경에서 실행합니다. 이것이 협업이나 자동화(CI) 상황에서 왜 편리할지 생각해 보세요.
	- uv를 통해서 재현성이 보장 되어, 어느 환경에서도 동일한 환경으로 실행이 가능하다.

## 6. 오늘의 한 줄 요약 (One-line takeaway)

> **`uv init` → `uv add` → `uv run`** 세 박자로 격리된 작업대를 차리고 패키지를 들이고 코드를 돌린다. 환경은 `pyproject.toml`+`uv.lock` 으로 자동 박제되어 어디서든 `uv sync` 로 재현된다. 그 위에서 NumPy 의 **벡터화** 한 줄이 앞으로의 모든 수치 연산의 출발점이 된다.
